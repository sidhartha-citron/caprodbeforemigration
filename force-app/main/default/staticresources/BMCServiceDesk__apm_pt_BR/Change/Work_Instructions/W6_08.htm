Work Instructions for Change Coordinators

6.8.1

When you are notified that work on the change request has been completed and if you cannot work on it, change the Status field of your task for updating the requesters and approvers of the change from Registered to Accepted. When you can update the requesters and approvers of the change request, set the Status field of the task to In Progress.

6.8.2

If the change implementation is not successful (for example, the change request is rejected or the change implementation had to be rolled back), add a Notes action in the change specifying the reason.

6.8.3

Send an email message to the approvers of the change request informing them whether the change is implemented successfully. If the change implementation was not successful, include the reason for the failure.

6.8.4

In the Linked Incidents section of the change request, open each request that was received from Request Management (if any). If the change implementation met the change requirements specified in the request, summarize how the change was implemented in the Resolution field of the request and set its Closure Category field to Resolved. If the change implementation did not meet the change requirements specified in the request, specify the reason for the failure in the Resolution field of the request and set its Closure Category field to Unable to solve or in conflict with standards or policy.

6.8.5

Set the Status field of the request to Completed to ensure that the service desk informs the person who is linked to the support request in the Client ID field.

6.8.6

In the Linked Problems section of the change request, open each problem that was received from Problem Management (if any). If the change implementation met the change requirements specified in a problem, add a Notes action summarizing how the change was implemented and set its Status field to Change Completed. If the change implementation did not meet the change requirements specified in a problem, add a Notes action indicating the reason for the failure. If the change was rejected by its approvers, set the Status field of such problems to Rejected or if the implementation of the change did not fulfill the requirements specified in a problem (for example, the implementation was only partially successful or it had to be rolled back), set the Status field to Change Completed.

Note:

Setting the Status field of a problem to Rejected or Change Completed causes a problem to be returned to its problem coordinator.

6.8.7

If the requirements for the change request were received from Release Management as a document, send an email message to the release coordinator to inform whether the change is implemented successfully. If the change implementation was not successful, include the reason for failure.

6.8.8

When you complete the update of the requesters and approvers, set the Status field of the task to Completed.

6.8.9

In the Duration (HH:MM) field, in the Notes action, enter the time that you spent completing the task.

Note:

If the implementation is partly successful and if a solution already exists for known issues, create a knowledge article for the provided solution.
