Work Instructions for Specialists

6.1.1

When the Status field of your task for testing the implemented change changes to Assigned, open it and read the instructions in the Description field and Notes actions.

6.1.2

If you are not ready to start working on the task, set the Status field of the task to Accepted.

Note:

If the task should be assigned to another person, ask the change coordinator to change its assignment. To do so, set the Status field of the task to Rejected, and add a Notes action saying why you rejected the task.

6.1.3

When you are ready to test the implemented change, set the Status field of the task to In Progress and perform a final test in the production environment to verify that all change requirements have been met.

Note:

If you do not have access rights to perform a test in the production environment, work with a user who has the access rights.

6.1.4

In the Resolution field of the task, describe the test and its result.

Note:

If you cannot continue to work on the task (for example, the production environment is temporarily inaccessible), set its Status field to Waiting for…, and add a Notes action saying what you are waiting for and when you expect to complete the task.

6.1.5

When you have completed the test and all change requirements have been fulfilled, set the Status field of the task to Completed.

If the implementation failed the test, determine whether the change implementation must be rolled back. In the Resolution field of the task, describe your conclusion and the reason for performing the production test. In this case, leave the Status field set to In Progress, and keep the task open while you continue with the next steps.

Note:

A change implementation must not be rolled back when it provides an improvement over the previous situation and does not cause a security or data integrity risk.

6.1.6

In the Duration (HH:MM) field, in the Notes action, enter the time that you have spent on the task.
