Work Instructions for Specialists

5.1.1

When the Status field of your task for the development of the new release changes to Assigned, open the task and read the instructions in the Description field and Notes actions.

5.1.2

If you cannot start working on the task, set the Status field of the task to Accepted.

Note:

If the task must be assigned to another developer, ask the change coordinator to change the task assignment. To do so, set the Status field of the task to Rejected, and add a Notes action saying why it has been rejected.

5.1.3

When you are ready to start developing the new release, set the Status field of the task to In Progress, and develop the new release.

Note:

If you cannot continue working on the task (for example, the development is more difficult than anticipated and you are waiting for help from a colleague), set the Status field of the task to Waiting for…, and add a Notes action saying what you are waiting for and when do you expect to complete the task.

5.1.4

When you complete the development of the new release, test the release to ensure that the change requirements have been met. Test in a separate test environment if one is available. Otherwise, perform your tests in the development environment. If (automated) scripts are available for regression testing for the application, update and run the scripts. Use the development environment to fix any errors that occurred during your tests. After you have confirmed that the change requirements have been met and no new issues have been introduced, specify which tests you have performed in the Resolution field of the task and set its Status field to Completed.

If the change requirements could not be met, set the Status field of the task to Failed, and specify in the Resolution field why it is not possible to develop the required functionality.

5.1.5

In the Duration (HH:MM) field, in the Notes action, enter the time that you spent completing the task.
