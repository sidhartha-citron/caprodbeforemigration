En la siguiente tabla se proporciona una lista con definiciones de una serie de términos y abreviaturas usados en el modelo de proceso de Alignability®.

Término

Definición

A

Alignability®

La capacidad de una organización proveedora de servicios de ajustar la funcionalidad de sus servicios y los niveles a los que estos se proporcionan con las necesidades de los clientes.

Disponibilidad

El porcentaje de horas de servicio durante las que la funcionalidad de un servicio es accesible a los clientes durante un lapso de tiempo.

Cambio de aplicación

Los siguientes casos se consideran cambios de aplicación:

  • Cuando se desarrollan aplicaciones nuevas

  • Cuando se ajusta la configuración o el código de aplicaciones existentes (esto incluye cualquier cambio al estándar de configuración de escritorio).

B

BIL

Nivel de importancia para el negocio (Business importance level, BIL)
La importancia de un servicio para el negocio del cliente. Generalmente, el nivel se determina mediante una estimación del impacto financiero de una indisponibilidad del servicio de una duración específica (por ejemplo, una semana).

El BIL se especifica en todos los SLA. Se usa para priorizar automáticamente las peticiones de resolución de incidencias una vez especificado el impacto.

La aplicación de gestión de servicios obtiene el BIL del SLA después de que el analista de servicio de asistencia haya seleccionado el usuario y el servicio para el que el usuario ha solicitado soporte.

Cuando, a continuación, el analista selecciona el nivel de impacto, la aplicación de gestión de servicios determina la prioridad de la petición de resolución de incidencias de la siguiente manera:

                 Impacto x BIL = Prioridad

Ejemplo:     Bajo x Crítico = P8

Error

En el contexto de gestión de servicios, se denomina error a un defecto en el software que provoca interrupciones de servicio.

Servicio de negocio

Un servicio proporcionado a un cliente.

C

CAB

Junta asesora de cambios (Change advisory board, CAB)
El grupo de personas responsable de mantener la funcionalidad y los niveles de servicio de un servicio específico alineados con las cambiantes necesidades empresariales de sus clientes.

La componen el propietario del servicio y los representantes de los clientes con un SLA activo para el servicio.

Elemento de catálogo

Una entrada en el catálogo de servicios de una organización proveedora de servicios. Un elemento de catálogo describe la funcionalidad del servicio, contiene un conjunto de objetivos de nivel de servicio (SLT) y especifica los cargos del servicio. Esto permite a los clientes saber qué pueden esperar cuando firmen un SLA basado en el elemento de catálogo.

CMDB

Base de datos de gestión de la configuración (Configuration management database, CMDB).
La base de datos en la que se registra y vincula la información de todos los componentes de infraestructura relevantes para proporcionar una descripción general precisa y completa de cómo debería estar configurada la infraestructura actualmente.

Cambio

Cuando se crea, añade, mueve, configura o elimina un CI.

Lo debe dirigir un coordinador de cambios cuando la implementación tiene los siguientes efectos:

  • que un servicio no esté disponible o se degrade durante horas de servicio

  • que cambie la funcionalidad de un servicio

  • que sea preciso actualizar la CMDB

Dentro del proceso de gestión de cambios, es importante distinguir dos tipos de cambio:

  • Cambio de infraestructura

  • Cambio de aplicación

Las siguientes categorías se aplican tanto para cambios de infraestructura como de aplicación:

  • Cambio estándar

  • Cambio no estándar

  • Cambio de emergencia

Cargo

El término "cargo" se usa dentro del modelo de proceso para indicar una suma de dinero que el cliente deberá pagar.

CI

Elemento de configuración (Configuration item, CI).
Un componente de hardware o software de la infraestructura de TI. Los certificados y documentos de licencia de software también se consideran elementos de configuración.

Queja

Una petición para que mejore el suministro de soporte.

Téngase en cuenta que estas peticiones se suelen enviar para ayudar a la organización proveedora de servicios a mejorar el soporte y siempre deben tratarse como contenido positivo.

Cumplido

Una expresión de elogio o ánimo por parte de un cliente.

Manual de continuidad

Un documento que indica el orden a seguir en la ejecución de planes de continuidad. El manual de continuidad también incluye toda la información de contacto y de sitio de continuidad relevante para la recuperación de los servicios.

Plan de continuidad

Un documento que brinda instrucciones técnicas detalladas sobre cómo una instancia de servicio específica se puede recuperar en su sitio de continuidad para seguir proporcionando el servicio desde allí. También proporciona instrucciones técnicas que se pueden seguir después de recuperar la instancia de servicio para volver del modo de continuidad al modo de producción normal.

Tabla de puntuación para evaluar riesgos de continuidad (Continuity Risk Assessment Scorecard)

Un formulario que se puede completar para ayudar a una organización cliente a determinar el nivel adecuado de cobertura de continuidad para una instancia de servicio que planea comenzar a usar. Este formulario también se puede usar cuando una organización cliente desea reevaluar su riesgo de continuidad para un servicio para el que ya tiene un SLA activo.

Sitio de continuidad

Una sede desde la que se pueden proporcionar uno o más servicios para evitar o resolver indisponibilidades de servicio. Puede actuar simultáneamente como lugar de producción de otros servicios.

Objetivo de continuidad

Un período durante el que una instancia de servicio debe recuperarse en su sitio de continuidad cuando el servicio que proporciona no está disponible a causa de un desastre.

Calendario de evaluación de continuidad

Un calendario en el que se programan las pruebas de recuperación para las instancias de servicio que tienen un SLA activo en el que se estipula un objetivo de continuidad.

Coste

El término "coste" se usa dentro del modelo de proceso para hacer referencia a una suma de dinero que deberá pagar la organización proveedora de servicios.

Cliente

Una organización o persona cubierta por una suscripción a un servicio (por ejemplo, un SLA) de la organización proveedora de servicios.

D

Desastre

En el contexto de gestión de servicios, un desastre es la imposibilidad de proporcionar servicios desde un lugar de producción durante un período prolongado a causa de uno de los siguientes motivos:

  • La destrucción (parcial) del lugar de producción

  • La pérdida de servicios de comunicación digital en el lugar de producción.

  • La pérdida de suministro eléctrico en el lugar de producción

  • La pérdida de climatización en el lugar de producción

  • La imposibilidad de acceder al lugar de producción.

DML

Biblioteca definitiva de medios (Definitive media library, DML)
Una o más ubicaciones seguras que, en conjunto, contienen las copias maestras de todas las versiones admitidas de los CI de software y los paquetes de distribución de software con sus respectivos certificados de licencia. También se incluyen en la DML los CI de software y los paquetes de distribución de software que ya no se admiten, pero deben permanecer disponibles en caso de que se requiera una reversión, con sus respectivos certificados de licencia.

E

Cambio de emergencia

Un tipo de cambio que se ha implementado de acuerdo con los procedimientos de gestión de cambios para la resolución de incidencias.

También se registra un cambio de emergencia por cada instancia de servicio que se restaura durante una recuperación de servicio. De esta manera se garantiza la actualización de la CMDB y la información de gestión de capacidad.

Escalar

Transferir la responsabilidad a un nivel superior en la jerarquía organizativa.

Evento

Una notificación generada automáticamente que advierte acerca de una incidencia actual o inminente.

F

Firmware

Programación almacenada en memorias programables de solo lectura. Suele usarse para controlar dispositivos de hardware como impresoras, módems, routers, etc. Ya que el firmware se crea, prueba, versiona y distribuye como software, no se diferencia entre software y firmware dentro de los procesos.

Funcionalidad

Las capacidades de un servicio. Lo que un servicio hace y/o permite que sus usuarios hagan.

H

Hardware

La parte física de los ordenadores y sus dispositivos relacionados.

I

TIC

Tecnologías de la información y la comunicación
Véase TI.

Aunque su significado sea idéntico al de "TI", se suele preferir este acrónimo porque abarca explícitamente la tecnología de servicios como correo electrónico, teléfono, videoconferencias, localizadores, etc.

Impacto

La medida en que el rendimiento o la funcionalidad de un servicio se degrada.

El impacto es mayor cuanto mayor sea la degradación de la funcionalidad del servicio y/o la cantidad de usuarios afectados.

La pérdida total de funcionalidad del servicio se conoce como indisponibilidad de servicio.

Incidencia

El período durante el que un servicio no funciona como debería. Las incidencias se conocen también como "interrupciones de servicio".

Véase Petición de resolución de incidencia.

Cambio de infraestructura

Los siguientes casos se consideran cambios de infraestructura:

  • cuando se instala, configura o quita un dispositivo de hardware

  • cuando se elimina software

  • cuando se realizan instalaciones adicionales de aplicaciones tras su desarrollo, prueba y aplicación iniciales

  • cuando se instala y configura software del sistema (por ejemplo, sistemas operativos, bases de datos, copias de seguridad de software, software antivirus, software de gestión de sistemas y redes, etc.).

  • cuando se ajustan programas de copias de seguridad o trabajos por lotes.

TI

Tecnologías de la información
Se refiere a todos los aspectos relacionados con el procesamiento computerizado de datos.

Véase TIC.

J

Trabajo

En el contexto de gestión de servicios, un trabajo es un conjunto de instrucciones que se pueden ejecutar automáticamente sin interacción por parte del usuario.

Los siguientes son ejemplos de tipos de trabajo:

Trabajo por lotes

Un programa informático que ejecuta una secuencia de comandos sin interactuar. Este tipo de trabajo se puede iniciar manualmente o se puede programar para que comience automáticamente en un momento dado y con una frecuencia específica.

Trabajo de copia de seguridad

Un conjunto de instrucciones para software de copia de seguridad. Estas instrucciones definen qué archivos copiar, dónde copiarlos, si se deben comprimir, etc. Los trabajos de copia de seguridad se pueden iniciar manualmente o se pueden programar para comenzar automáticamente en un momento dado y con una frecuencia específica.

Trabajo de monitorización

Un conjunto de instrucciones para una aplicación de gestión de redes o sistemas. Estas instrucciones definen qué se necesita monitorizar, el intervalo de sondeo, cuándo se debe generar un evento, etc.

K

Error conocido

Un problema del que se conoce la causa origen.

KPI

Indicador del rendimiento clave (Key performance indicator, KPI)
Un resultado fundamental y evaluable que permite hacer un seguimiento de la eficacia, efectividad y previsibilidad de un proceso.

M

Incidencia grave

Una incidencia con un impacto negativo serio en el negocio del cliente. El cliente tiene un SLA para el servicio impactado con un BIL "Crítico" o "Core (Principal)".

Equipo de incidencia grave

Un equipo predefinido compuesto por las personas involucradas en la resolución de una incidencia grave.

Encargado del equipo de incidencia grave

La persona responsable de coordinar el esfuezo de resolución de una incidencia grave.

N

Cambio no estándar

Un tipo de cambio que debe coordinar un coordinador de cambios y para el que no hay una plantilla aprobada disponible.

Lo opuesto a un cambio no estándar es un cambio estándar.

O

Disposición operativa

La disposición operativa se define por la presencia de los siguientes factores:

  • Comunidad de usuarios

  • Organización proveedora de servicios

  • Entorno de producción

  • Entorno de continuidad (cuando corresponda)

para aceptar un cambio.

P

Rendimiento

La velocidad a la que un servicio ejecuta transacciones.

El rendimiento se suele medir desde el momento en que un usuario da el comando de ejecución para una transacción (por ejemplo, cuando hace clic en el botón Guardar) hasta que recibe la respuesta que indica que la acción se ha completado (por ejemplo, hasta que pueda enviar otra transacción).

Cambio programado

Un cambio para el que un coordinador de cambios ha preparado un plan de implementación antes de implementarlo.

Este cambio puede ser un cambio estándar o un cambio no estándar.

Prioridad

El grado de urgencia con el que una petición de soporte debe completarse.

Problema

Interrupción de servicio recurrente

Tenga en cuenta que una única incidencia ya puede conducir a la identificación de un problema cuando se espera que la interrupción del servicio se repita. Es posible identificar un problema incluso antes de que ocurra una incidencia (por ejemplo, cuando la capacidad de una instancia de servicio está a punto de agotarse).

Lugar de producción

Una instalación que, en circunstancias normales, se usa para proporcionar uno o más servicios. Puede ser, al mismo tiempo, el sitio de continuidad de otros servicios.

R

Sala de control de recuperación

La ubicación desde la que se asignan y coordinan las tareas de recuperación del servicio. Es el punto de información para las actualizaciones de progreso durante una recuperación de servicio.

Se encuentran dentro o cerca de cada uno de los sitios de continuidad.

Plazo de decisión de recuperación

El punto temporal, tras la indisponibilidad del primer servicio a causa de un desastre, antes del que se debe llamar a los equipos de recuperación para comenzar la recuperación del servicio.

Prueba de regresión

La prueba de un nuevo despliegue para garantizar que la funcionalidad de la versión anterior de la aplicación siga vigente.

Incidencia relacionada

Una incidencia con un servicio complementario que causa la incidencia actual.

Despliegue

Un conjunto de cambios para la resolución de una o más peticiones de cambio no estándar.
También puede tratarse de la versión de una aplicación que está lista para ser transferida a su entorno de prueba o producción.

Fiabilidad

La cantidad de veces que la funcionalidad de un servicio no estuvo disponible para sus clientes durante horas de servicio en un período determinado.

Petición de cambio

Una petición para añadir, modificar, mover o quitar hardware, software o datos.

Tenga en cuenta que las peticiones de copia de datos (por ejemplo, para hacer una copia de seguridad o restaurar una copia de seguridad en una ubicación diferente) y las peticiones de trabajos por lotes también entran en esta categoría.

Petición de resolución de incidencia

Una petición para reparar un servicio que no funciona como debería.

Tenga en cuenta que también entran en esta categoría las peticiones de restablecimiento de contraseña por parte de usuarios que no la recuerdan y las peticiones de restauración de copias de seguridad de usuarios que han perdido datos.

Véase Incidencia.

Resolución de peticiones

La solicitud de respuesta a una pregunta relacionada con el servicio o una petición que se puede implementar siguiendo un flujo de trabajo predefinido cuando:

  • Existe una plantilla para el flujo de trabajo.

  • No es necesario un análisis de riesgos e impacto.

  • No se necesita más aprobación que la del cliente.

Reversión

La restauración del entorno original cuando fracasa la implementación de un cambio o despliegue.
También se refiere a la cancelación de una transacción de datos.

Causa origen

La causa fundamental de un problema, cuya eliminación prevendrá que se repitan las incidencias resultantes del problema.

Análisis de causa origen

La identificación de la causa fundamental de un problema y la propuesta de una solución estructural.

S

Servicio

En el contexto de gestión de servicios, se define como la agrupación lógica de funcionalidad disponible a través de la combinación y configuración específica de CI de software y hardware .

Catálogo de servicios

Una descripción general de todos los servicios que una organización proveedora de servicios ofrece a sus clientes. Por cada servicio, el catálogo incluye uno o más elementos de servicio.

Degradación de servicio

Un servicio se degrada cuando algunas de sus funcionalidades no funcionan correctamente o cuando su rendimiento es lento.

Servicio de asistencia

Un grupo de personas dentro de una organización proveedora de servicios con las que los usuarios se pueden poner en contacto para obtener soporte para los servicios proporcionados por la organización.

Interrupción del servicio

Véase incidencia.

Horas de servicio

El horario durante el que el servicio debe estar disponible.

Instancia de servicio

Una instancia de servicio es un entorno que hace que la funcionalidad de un servicio esté disponible. Pueden existir numerosos entornos para proporcionar, en esencia, la misma funcionalidad, pero con distintos objetivos y/o para distintos usuarios. Por ejemplo, se pueden establecer dos instancias de servicio para el servicio Finanzas. Los empleados de la organización que se encuentran en Asia usan una instancia y los que se encuentran en Europa, la otra. Al vincular a una incidencia una instancia de servicio (en lugar de un servicio), RemedyForce puede calcular con precisión el nivel real de servicio que se ha proporcionado a cada cliente. Por ejemplo, cuando una incidencia afecta la instancia de servicio Finanzas en Europa, es importante que esta incidencia no se incluya en los informes de gestión de nivel de servicio para los clientes en Asia del servicio Finanzas. Las instancias de servicio también se usan para diferenciar entre el entorno de producción de un servicio y su desarrollo, y los entornos de prueba. Esto permite que las incidencias que afectan a los entornos de desarrollo y prueba se excluyan de los informes de gestión de nivel de servicio que se preparan para los clientes que solo usan el entorno de producción. Una instancia de servicio puede componerse de varios CI (por ejemplo, una aplicación y un software de servidor web) y varias instancias de servicios secundarias (por ejemplo, una instancia de servicio de Linux y una instancia de servicio de base de datos). A su vez, una instancia de servicio secundaria también puede constar de varios elementos de configuración (por ejemplo, un servidor Linux y un sistema operativo Linux) y varias instancias de servicio secundarias (por ejemplo, una instancia de servicio de almacenamiento y una instancia de servicio de espacio en bastidores). Se debe establecer un SLA por cada cliente que usa la instancia de servicio.

Indisponibilidad de servicio

Se considera que un servicio no funciona o no está disponible cuando ninguna de sus funcionalidades están disponibles.

Lista de comprobación de evaluación de situación

Un formulario que ayuda al gestor de guardia a tomar las decisiones correctas cuando considera cómo procesar un desastre (inminente). En especial, garantiza que se tengan en cuenta los siguientes aspectos:

  • ¿Se han perdido o peligran vidas humanas?

  • ¿Qué instancias de servicio se vieron perjudicadas o se espera que sufran el impacto?

  • ¿Está alguna de estas instancias de servicio cubierta por un SLA activo que estipula un objetivo de continuidad?

  • ¿Cuándo comenzó la indisponibilidad del primer servicio o cuándo se espera que el primer servicio deje de estar disponible?

  • ¿Se requiere alguna acción especial a causa de la naturaleza del desastre?

  • ¿Se puede resolver la situación iniciando una recuperación del servicio?

  • ¿Se ha cumplido el plazo de decisión de recuperación?

SLA

Acuerdo de nivel de servicio (Service level agreement, SLA)
El acuerdo entre la organización proveedora de servicios y el cliente del servicio.

Se debe establecer un SLA por cada combinación de una instancia de servicio y un cliente que la usa.

SLT

Objetivo de nivel de servicio (Service level target, SLT)
El nivel de servicio mínimo, definido en los SLA, que la organización proveedora de servicios se ha comprometido a proporcionar al cliente del SLA.

No tiene sentido establecer un SLT si no se mide el nivel de servicio real de modo que se puedan comparar ambos niveles.

SLR

Requisitos de nivel de servicio (Service level requirement, SLR)
El nivel mínimo de servicio requerido por un cliente, según lo define el gestor de nivel de servicio a partir de los requisitos empresariales proporcionados por el cliente.

El gestor de disponibilidad del servicio traduce los SLR en una propuesta de un conjunto de SLT. Los SLT propuestos conforman los niveles mínimos de servicio que la organización proveedora de servicios se compromete a proporcionar al cliente. El conjunto de SLT propuesto tiene una relación directa con el cargo que se le cobrará al cliente por el servicio.

Software

La programación que se usa para controlar ordenadores y sus dispositivos relacionados.

Cambio estándar

Un tipo de cambio que debe coordinar un coordinador de cambios y para el que existe una plantilla de cambios aprobada por el propietario del servicio.

La plantilla de cambios incluye un conjunto de órdenes de trabajo que define el flujo de trabajo que se debe seguir para la implementación de este tipo de cambio.

Lo opuesto a un cambio estándar es un cambio no estándar.

Soporte

Las actividades llevadas a cabo para garantizar que la funcionalidad de los servicios se proporcione a un nivel que cumpla con los SLT especificados en los SLA entre la organización proveedora de servicios y sus clientes.

Horas de soporte

El horario durante el que se debe proporcionar soporte para el servicio.

Petición de soporte

Solicitud de un usuariopara recibir soporte de la organización proveedora del servicio.

Los especialistas pueden enviar las peticiones de soporte al servicio de asistencia en representación de lo usuarios. Asimismo, los operadores pueden registrar las peticiones de soporte en representación de los usuarios.

Cambio soportado

Un tipo de cambio que se define en un elemento de catálogo. La organización proveedora de servicios acepta implementar un cambio soportado cuando un cliente, con un SLA activo basado en un elemento de catálogo que lo incluye, envía una petición de implementación.

T

Tarea

El término "tarea" se usa dentro del modelo de proceso para hacer referencia a una petición de soporte, a un problema o a una orden de trabajo.

Servicio técnico

Un servicio que no se proporciona al cliente directamente, pero que el proveedor del servicio necesita para proporcionar otros servicios.

U

Usuario

Una persona que ocasionalmente usa uno o más CI y/o servicios proporcionados por la organización proveedora de servicios.

V

Verificación

La acción de determinar si se ha cumplido un conjunto de requisitos establecidos previamente.

W

Formulario de petición web

Un formulario en el sitio de Internet o Intranet de la organización proveedora de servicios que los usuarios y/o representantes del cliente pueden completar y enviar para solicitar que se ejecute un cambio soportado.

Cada formulario de petición web solicita que se complete información específica para que se ejecute la petición. Una vez que se envíe un formulario de petición, se genera automáticamente una petición de soporte en la aplicación de gestión de servicios. Esta petición de soporte se asigna al grupo responsable de la ejecución de la tarea que se puede solicitar con el formulario usado para generar la petición. La petición de soporte contiene la información con la que se completó el formulario de petición web.

Orden de trabajo

Una tarea vinculada a un cambio. Constituye un paso dentro de un plan de implementación de cambios.

Solución provisional

Una solución temporal que omite o enmascara el funcionamiento incorrecto de un servicio. Se implementa cuando es la forma más rápida de permitir que los usuarios afectados reanuden su trabajo.
