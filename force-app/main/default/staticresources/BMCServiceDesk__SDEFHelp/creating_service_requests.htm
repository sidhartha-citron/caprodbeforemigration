You can submit a service request on behalf of clients when the clients cannot create their own service requests. Clients might be traveling and unable to access Self Service, or clients might call you and ask you to submit a service request for them. When you submit a service request for a client, the client can view the service request in Self Service to follow its approval history and the actions that you have performed on the service request. You can also create service requests for yourself and other staff members.

Note: Create service requests when a client is requesting a service from the service desk. For example, request for a mobile phone or a new laptop. Create incidents when client reports an issue. For example, if the mobile phone or laptop of a client is not working or a client is having an issue when using the mobile phone or laptop.

BMC Remedyforce provides the AutoComplete feature for the First Name, Last Name, Client ID, Category, Impact, and Urgency fields in this form.

To create a service request in Visualforce page layout UI

  1. In the BMC Remedyforce home page, click Workspaces.
  2. From the list of Workspaces, expand Incident Management.
  3. In the Incident Management list, click Incidents.
  4. In the incidents list view, click New to open a new incident form.
  5. Click the Last Name list.

    6. Depending on how system administrator has configured the settings, the Select From Clients or Search Clients window is displayed. If your system administrator has configured Application Settings to allow you to create incidents for contacts and leads, the Search Clients window is displayed where you can search for a user, contact, or lead. A list of matching entries is displayed under separate tabs of Users, Contacts, and Leads.

    If your system administrator has configured Application Settings to prevent you from creating incidents on behalf of contacts and leads, the Select From Clients window is displayed with a list of users. Double-click a record to select it.

    The type of client, user, contact, or lead, is displayed in the incidents list view in the Client Type and Client Name columns. The Client ID column in the incidents list view is blank if the incident record is created for a lead or a contact.

  6. In the Search Clients window, in the Keywords field, enter the search keywords, and press Enter or click .

    7. The list of matching records is categorized under Users, Contacts, and Leads (if applicable). You can change the number of records displayed in the list in the Maximum Records field. A maximum of 200 records per object can be displayed.

    The data fields, such as Account ID, Mobile Phone, Status, and so on, that display data of the matching records, can be configured on the Global Search Settings page. Contact your system administrator to configure the fields that you want to view in the search results.

    The following table describes which clients appear in the Select From Clients or Search Clients window depending on the fields that you have populated for the service request:

    Populated field

    The Select From Clients window displays the clients of the selected account. You can view and select other service request definitions by clearing the Account check box in the Views menu, and clicking Apply.

  7. Double-click the appropriate record.

    8. The First Name, Client ID, and Account fields are filled with the first name, client ID, and account (if exists) of the selected client. You can change the account of the client.

    The icon displays the same fields that are displayed in the Search Clients window, such as Account ID, Mobile Phone, and so on of the selected record in a pop-up window.     9. In some cases, you might have to create the service request for the parent account of the client's account when the applicable Service Level Agreement (SLA) exists with the parent account. For example, when the client's account is Pre-Sales and you have an SLA with the parent account of the Pre-Sales account (Sales account), you select the Sales account in the Account list.

  9. Click the Category list.
  10. In the Select From Categories window, perform the following actions:
    1. Select the category of the service request.

      2. Categories allow you to classify the service request using standard classifications, which can be tracked for reporting purposes.

      If you click the Category list after selecting a service request definition in the Request Definition list, the Select From Categories window displays service request categories and categories that are available to both Incident Management and Service Request Management. Incident categories appear in this window if they are parent categories to service request categories. If you want to select an incident category for the service request, select Filter Options > Incident Categories.

    2. To use the standard description of the selected category, select the Standard Description check box.

      3. The standard description of the selected category is appended to the Description field of the incident form.

    3. To use the standard resolution of the selected category, select the Standard Resolution check box.

      4. The standard resolution of the selected category is appended to the Resolution field of the incident form.

    4. Click OK.

    When you select a service request definition in the Request Definition list, the category, service, and service offering of the service request definition are populated in the Category, Service, and Service Offering fields of the Incident form. You can select different values in these fields after you select a service request definition. If the Category, Service, and Service Offering fields of the Incident form are populated before you selected a service request definition, the category, service, and service offering of the service request definition replaces the existing values in the Category, Service, and Service Offering fields.

    14. By default, the following service request fields appear in the Service Request Detail window:

    • Requested By
    • Requested For
    • Email, Phone
    • Quantity
    • Unit Price
    • Total Price
    • Date Required
    • Date Expected

      • In addition to these fields, the system administrator can configure the service request to display the following fields:

    • Documentation
    • Instructions
    • View a Request
    • Assigned To
    • Approvals

      • If you type an increased quantity in the Quantity field, the Total Price field is updated to display the total price of the service request. For example, if the Unit Price of the service request is $100 and you type 5 as the new quantity in the Quantity field, the Total Price field displays $500.

  14. Enter values in the input fields to provide the information that is used to fulfill the service request.

    15. Note: You must enter values in the mandatory fields of a service request before you submit a service request for fulfillment.

  15. Click Save.

    16. After you save the service request in the Service Request Detail window, you cannot update the values that you have entered in the service request fields. If you want to update the values, you must create a new service request from the same service request definition. To create a new service request from the same service request definition, close the Service Request Detail window, select the same service request definition from the Request Definition list, and enter the updated values.

  16. Click Close in the upper-left section of the Service Request Detail window to return to the incident form.
  17. Depending on the client requirements, complete adding values to the remaining fields of the service request.

    18. For more information, see Creating an incident in the Visualforce page layout UI.

  18. (Optional) If you want to assign a different service and service offering to service request instead of using the service and service offering of the service request definition, enter the following information in the Service and CI Details section:
    1. Service—Select the required service that is associated with the current service request.

      2. When you click this field, the Select From Business Services window appears. This window displays services of the following types:

      • Unknown
      • Business Service
      • Technical Service

        • In this window, you cannot select service offerings as the service that is associated with the current service request.

        When you select a service, BMC Remedyforce applies all SLAs that match the service and the account of the service request. You can view the service target transactions that occur when an SLA is applied to the service request by clicking .

    2. Service Offering—Select the required service offering of the service that is selected in the Service field.

      3. When you click this field, the Select From Business Services window appears. This window displays services of the Offering type only.

      If you select a service offering in the Service Offering field without selecting a service in the Service field, the service of the selected service offering is populated in the Service field when you click Save.

  19. Click Save.

    20. After you save the service request, you can only view the values that you have entered in the service request fields by clicking the Service Request Detail icon next to the Request Definition list. If you want to update the values, you must create a new service request.

    The Supporting Information section is not loaded in a new form. When you save the record, the Show Supporting Information link appears. Click this link to view the Supporting Information section. You can configure the Supporting Information section to always display by selecting the Always show Supporting Information for existing records check box.

    If the selected service request is configured to be submitted for approval when matching criteria is met, the required approvals are created in the Approvals tab of the Supporting Information section. When the service request is approved, BMC Remedyforce creates the change request and task records of the processes that you have configured for the service request. If the selected service request is configured to be manually submitted for approval, BMC Remedyforce creates the change request and task records of the processes when the service request is submitted.

    BMC Remedyforce links the change request and task records to the service request. When you open the change request and task records, the Service Request # link appears in the Details section of these records. If the logged-on user does not have access to the service request, the Service Request # link does not appear.

    Depending on the mapping between the Input fields and the fields of the processes that are configured for the service request, the values of the Input fields are populated in the service request, change request, and task records that are created. If no mappings are created, the process values are populated in the service request, change request, and task records. When mappings are created, the values of the Input fields override the process values. These change request and task records must be completed to fulfill the service request. BMC Remedyforce creates a Process Execution Failure action record in the History tab of the Supporting Information section when an error occurs while creating the change and task records of the processes. The error might occur because an approval process is not set up, the approval criteria are not met, or an approver is not defined.

    The Automatically Submit for Approval (when matching criteria is met) check box must be selected for the service request to be submitted for approval when matching criteria is met. When you copy a service request that is created from a service request definition with the Automatically Submit for Approval (when matching criteria is met) check box selected, the copied service request is not submitted for approval. You must manually submit the service request for approval.

Error when saving the service request

If the logged-on user is a staff member or a user of the Change Manager profile and the logged-on user creates a service request and saves the service request, an Insufficient access on cross reference entity error occurs if the logged-on user does not have permissions to the service request. This error occurs in the following cases:

    • Case 1: When the logged-on user selects the following while creating the service request:
      • A client who is different from the logged-on user.
      • A service request definition where the process contains the owner ID of a user who is higher in the role hierarchy or the owner ID of a user whose records cannot be seen by the logged-on user.
    • Case 2: When the logged-on user selects the following while creating the service request and assigns the service request:
      • A client who is different from the logged-on user.
      • The logged-on user assigns the service request to a user who is higher in the role hierarchy or to a user whose records cannot be seen by the logged-on user.

        • To prevent this error from occurring, logged-on users must assign the service request to themselves before saving the service request. After saving the service request, logged-on users can assign the service request to the required users.
