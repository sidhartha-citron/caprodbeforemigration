If an incident is not assigned to you, you can edit it only if you meet any of the following conditions:

    • The incident is assigned to a staff member who is lower in the role hierarchy than you are, and you have read access to the Incident custom object.
    • The incident is assigned to a queue and you are a member of this queue.
    • The staff member assigned to this incident has shared the incident record with you.
    • The system administrator has changed the default sharing access of incidents from the out-of-the-box access level of Public Read Only to Public Read/Write.

Only system administrators can edit the Priority, Closed Date, and Responded Date fields of an incident.

In console layouts, to revert the changes that you make to an incident before saving the record, click Reload.

Note: In the Visualforce page layout UI, to erase the updates that you have done to the record and revert the record to the earlier saved state, click Refresh.

Tracking changes

When you edit an incident in Remedyforce Console, Incident Console or Incidents tab, the changes are tracked in the Incident History section. When you edit an incident in the Visualforce page layout UI, the changes are tracked in the Auditing tab in the Supporting Information section. All fields on the incident form are not tracked.

By default, Salesforce allows you to track 20 fields on a form. The following fields on the incident form are tracked by default:

All Tasks Closed Controller

Due Date

Priority

Call Count

Impact

Record Update Count

Category

Outage End

Status

Client ID

Outage Start

Urgency

To record the date and time when records are updated, BMC recommends that you always track the Record Update Count field. For all the fields of the object, this field helps you track the user who makes the change and the time when the record is changed.

In the Visualforce page layout UI, the information about the changes is displayed in the form of a table (Created Date, Created By, Changed Field, Old Value, New Value).

If your administrator is tracking the Record Update Count field and you modify a tracked field, two records are created in the Incident History section. One record specifies the change and the other record displays the number of times the incident is modified.

For example, you have changed the status of the incident from Opened to Pending. The following messages appear:

Changed Status from Opened to Pending.
Changed Record Update Count from <x> to <y>.

If your system administrator is not tracking the Record Update Count field, the following message does not appear in the Incident History section: 

Changed Record Update Count from <x> to <y>.

In Visualforce page layout UI, the information about the changes is displayed in the form of a table (Created Date, Created By, Changed Field, Old Value, New Value).

When you modify a field that is not being tracked and your system administrator has enabled tracking for the Record Update Count field, the following message appears in the Incident History section:

Changed Record Update Count from <x> to <y>.

Editing multiple incidents

  1. Click the Remedyforce Console tab.
  2. From the View menu, select Incidents/Service Requests.
  3. Select the check boxes for the records that you want to edit, and click Update.

    4. You can update only 200 records at the same time.

  4. From the Field list, select the field that you want to update for the selected records.

    5. Only the fields that are shown on the list view are available in the Field list. Fields of type Auto Number and Formula are not available in the Field list. If you do not have permission to edit any of the fields in the list view, the field is not available in the Field list.

  5. Enter or select a value for the selected field.
  6. Click Apply.

    7. You can update one field of multiple records at the same time.

  7. To update another field of multiple records, repeat step 4 to step 6.
  8. Click OK.

Due Date recalculation

The due date calculation options are available in the Due Date Calculation window when you are editing a record. The Due Date Calculation window is displayed if your system administrator has defined the following settings:

    • The When the Priority changes, recalculate the Due Date without user input check box is clear. This check box is provided on the General Application Settings page.

      • If check box is selected, the Due Date Calculation window is not displayed and the due date is recalculated based on the Use Opened Date instead of Current Date to recalculate Due Date setting.

    • The Open popup dialog for recalculating due date when priority of incident changes workflow is active.

When you are editing a record and you update the impact or urgency that leads to a change in the priority of the record, the Due Date Calculation window appears. The following options are available on the Due Date Calculation window:

    • Do not recalculate—The existing due date is retained for the record.
    • Recalculate based on open date—The due date is recalculated by using the opened date. For example, if the priority is changed to 1 and the duration assigned to a priority of 1 is 2 hours, the recalculated value of the due date is 2 hours added to the date and time when the incident was created.
    • Recalculate based on current date—The due date is recalculated by using the current date. For example, if the priority is changed to 1 and the duration assigned to a priority of 1 is 2 hours, the recalculated value of the due date is 2 hours added to the current date and time.
