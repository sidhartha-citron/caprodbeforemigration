A change request allows you to implement a controlled process for the addition, modification, or removal of approved and supported hardware, networks, software, applications, environments, or systems. The change request records the detailed information about the change, such as the reason for the change, the rollout plan, the back-out plan, the type of change, and the change category. You can create a pre-approved change requests also.

BMC Remedyforce provides the AutoComplete feature for the Initiator First Name, Initiator Last Name, Initiator ID, Category, Impact, and Urgency fields in the Change Request form.

To create a change request

  1. In the BMC Remedyforce home page, click Workspaces.
  2. From the list of Workspaces, expand Change Management.
  3. In the Change Management list, click Change Requests.
  4. In the change requests list view, click New.
  5. Click Template.

    6. To help ensure accuracy and completeness, BMC recommends that you use a template to help create the change request. A template ensures consistency in the way information about a change request is captured. Your administrator can define templates for commonly occurring change requests.

  6. In the Select from Templates window, double-click an applicable template.

    7. The values of the fields in the template are filled in the corresponding fields of the Change Request form.

  7. Click the Initiator Last Name list.
  8. In the Select from Clients window, select the initiator who contacted the service desk with the change request.

    9. The Initiator First Name and Initiator ID fields are filled with the first name and Initiator ID of the selected initiator.

  9. Click the Category list.
  10. In the Select from Categories window, perform the following actions:
    1. Select the category of the change request.

      2. Categories allow you to classify the change request using standard classifications, which can be tracked for reporting purposes.

    2. Click OK.
  11. Type a description of the change request that you want to implement in the Description field.

    12. You can use the Documentation tab in the Supporting Information section to attach a file or document that provides detailed information about the change request that you want to implement.

  12. Click Reason for Change.
  13. In the Reason for Change field, type the reason why you are creating a change request.

    14. You can also use the Documentation tab in the Supporting Information section to attach a file or document that provides a detailed explanation of the need for the change request.

  14. Click Rollout Plan.
  15. In the Rollout Plan field, type the procedures for implementing the change request reason.
  16. Click Back-out Plan.
  17. In the Back-out Plan field, type the back-out plan for restoring the configuration item to its initial state before the change request was implemented.
  18. After the change request is approved, select the Approved check box.

    19. After selecting this check box, you can write a workflow that populates the template of the change request to update the status to Approved and create necessary tasks.

  19. From the Review Date field, select the date when the change request is scheduled to be reviewed after the change request is closed.

    20. The review process is necessary to evaluate the success or failure of the change request after a certain time has elapsed.

  20. In the Review Comments field, type any comments you want to track with the change request based on your review.

    21. These notes might include an assessment of the success of the change request or reasons why the change request failed and how the failure might be avoided in future.

  21. In the Note field, type notes about the change request.

    22. The change managers can type updates about the change request in the Note field and click the Add icon. These updates are added to the Supporting Information section as history records in the History tab. The Note field allows change managers to add notes about the change request without having to navigate to the Supporting Information section and create a history record.

    Note: The Note and Duration fields are enabled only after you save the record.

  22. In the Duration field, type the amount of time that you spent working on the change request.
  23. Click Add.

    24. The Supporting Information section is displayed. The contents of the Note and Duration fields are added to the Supporting Information section as a history record in the History tab and the Supporting Information section with the history record is displayed.

  24. In the Details section, enter the following information:
    1. Status—Select the status that you want to assign to the change request.

      2. You can track the progress of a change request from its stages of opening to closure by using this field.

      The Change Request Progression Bar in the header of the Change Request form is updated depending on the stage that is linked to the status you have selected. For example, if the Working status is linked to the In Process stage and you select Working as the status of the change request, the In Process stage is selected in the Change Request Progression Bar.

    2. Change Type—Select the type of change (Standard, Normal, or Emergency).
    3. Change Category—Select the category of change (Minor, Significant, or Major).
    4. Impact—Select the impact that you want to assign to the change request.

      5. The level of impact and the urgency are used to calculate the priority of the change request.

    5. Urgency—Select the urgency that you want to assign to the change request.

      6. The urgency and the level of impact are used to calculate the priority of the change request.

    6. Priority—When you select an urgency and impact and save the record, the Priority field is filled in. The priority is calculated depending on the urgency and impact values that you have selected.
    7. Opened Date—This field is filled with the date and time at which the change request is created.
    8. Responded Date—This field is filled with the date and time when you respond to the change request by using the Actions > Responded menu command.
    9. Scheduled Start—Select the date when the change request is scheduled to be implemented.
    10. Scheduled End—Select the date when the change request must be completed.
    11. Actual Start Date—Select the date when you start implementing the change request.

      12. This field tracks change requests that were not started on the scheduled start date.

    12. Actual End Date—Select the date when the change request is completed.

      13. This field tracks change requests that were not completed before the scheduled end date.

    13. Closed Date—This field is filled with the date and time at which the change request is closed.
    14. Service Request #—This field displays the record number of the associated service request, if any. The record number is displayed as a hyperlink to the appropriate record. When you click the record number, the record is displayed in the corresponding tab. This field is displayed only if the change request is linked to a service request. You can also copy the record number of the linked service request.
  25. In the Custom Fields section, enter values for the custom fields that are added by the system administrator.

    26. Note: If you are using the custom form and your system administrator has added fields from the out-of-the-box field sets, enter values for the fields present on the custom form.

  26. Click Save.
  27. After you have created the change request, you can add supporting information for the change request.

    28. For more information, see Supporting information for change requests.

  28. (Optional) Select the Inactive check box to designate the current record as inactive.

    29. You might need to make a record inactive because the record is no longer valid, but must be retained for compliance with the records retention policy. For example, if a configuration item has been damaged beyond repair because of a power outage, all change requests for the configuration item are invalid.

    Note: Records marked as inactive are not included in any search performed on the database.

Supporting information for change requests

The Supporting Information section is not loaded in a new form. When you save the record, the Show Supporting Information link appears. Click this link to view the Supporting Information section. You can configure the Supporting Information section to always display by selecting the Always show Supporting Information for existing records check box. To hide the Supporting Information section, click the Hide link.

The Supporting Information tabs at the lower section of the Change Request form allow you to:

    • Track the history of the change request
    • Link problems, services, configuration items, and incidents
    • Add new change assessments
    • Submit for approval or recall the approval
    • Add documentation
    • Add new tasks
    • View audit information about the change request

The Change Request form contains the following Supporting Information tabs:

    • History—Allows you to track the history of all actions that have been performed on the change request by viewing the existing user-defined and system-defined actions or adding new user-defined actions.
    • Problems—Allows you to link existing problems that are the source of the change request.
    • Services and CIs—Allows you to link existing services and configuration items that are affected when you implement the change request.
    • Assessments—Allows you to add new change assessments that must be performed to before you approve the change request.
    • Approvals—Allows you to submit the change request for approval and recall a change approval.
    • Tasks—Allows you to add new tasks that must be performed to complete the change request.
    • Incidents—Allows you to link existing incidents that are the source of the change request.
    • Documentation—Allows you to attach documents and URLs to the attachments that are required to close the change request.
    • Auditing—Displays the history of changes made to fields in the form. The BMC Remedyforce system administrator enables field history tracking for required fields.
