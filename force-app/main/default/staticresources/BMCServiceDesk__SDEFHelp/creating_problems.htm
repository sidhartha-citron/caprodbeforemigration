This section provides steps to create a problem.

To create a problem in the Visualforce page layout UI

  1. In the BMC Remedyforce home page, click Workspaces.
  2. From the list of Workspaces, expand Problem Management.
  3. In the Problem Management list, click Problems.
  4. In the problems list view, click New.
  5. From the Problem Source list, select the source of the problem.
  6. Click the Category list.
  7. In the Select from Categories window, perform the following actions:
    1. Select the category of the problem.

      2. Categories allow you to classify the problem using standard classifications, which can be tracked for reporting purposes.

    2. Select the Standard Description check box to use the standard description of the selected category.

      3. The standard description of the selected category is populated in the Description field of the problem form. The standard description replaces the description that you have typed in the Description field.

    3. Select the Standard Resolution check box to use the standard resolution of the selected category.

      4. The standard resolution of the selected category is appended to the Resolution field of the problem form. The standard resolution replaces the resolution that you have typed in the Resolution field.

    4. Click OK.
  8. If a problem is a known error, perform the following actions:
    1. Select the Known Error check box.

      2. Note: When you select this check box, the Workaround button is enabled and you can enter a workaround for the known error.

    2. Click Workaround.
    3. Type the workaround for the known error.
  9. In the Description field, type a description of the problem that you want to resolve.

    10. After saving the record, you can attach a file or document that provides detailed information about the problem that you want to resolve in the Documentation tab of the Supporting Information section.

  10. Click Resolution.
  11. Type the resolution that must be implemented for this problem in the Resolution field.
  12. In the Note field, type notes about the problem.

    13. The staff members can type updates about the problem in the Note field, and click the Add icon. These updates are added to the Supporting Information section as history records in the History tab. The Note field allows staff members to add notes about the problem without having to navigate to the Supporting Information section and create a history record.

    Note: The Note and Duration fields are enabled only after you save the record.

  13. In the Duration field, type the amount of time that you spent working on the problem.
  14. Click Add.

    15. The Supporting Information section is displayed. The contents of the Note and Duration fields are added to the Supporting Information section as a history record in the History tab and the Supporting Information section with the history record is displayed.

  15. In the Details section, enter the following information:
    1. Status—Select the status that you want to assign to the problem.

      2. You can track the progress of a problem from its stages of opening to closure by using this field.

      The Problem Progression Bar in the header of the problem form is updated depending on the stage that is linked to the status that you have selected. For example, if the Working status is linked to the In Process stage and you select Working as the status of the problem, the In Process stage is selected in the Problem Progression Bar.

    2. Impact—Select the impact that you want to assign to the current problem.

      3. The level of impact and urgency are used to calculate the priority of the problem.

    3. Urgency—Select the urgency that you want to assign to the current problem.

      4. The urgency and the level of impact are used to calculate the priority of the problem.

    4. Priority—When you select an urgency and impact and save the record, the Priority field is filled in. The priority is calculated depending on the urgency and impact values that you have selected.
    5. Opened—This field is filled with the date and time when a problem is created.
    6. Due Date—When you select an urgency and impact and save the record, the Due Date field is filled in. The due date is calculated based on the duration of the priority.

      7. If required, you can select a different due date from the calculated due date.

      When you are editing a problem and you update the impact or urgency that leads to a change in the priority of the problem, the following Due Date Calculation options appear:

      • Do not Recalculate—The existing due date is retained for the problem.
      • Recalculate Based on Open Date—The due date is recalculated by using the open date. For example, if the priority is changed to 1 and the duration assigned to a priority of 1 is 2 hours, the recalculated value of the due date is 2 hours added to the date and time when the problem was created.
      • Recalculate Based on Current Date—The due date is recalculated by using the current date. For example, if the priority is changed to 1 and the duration assigned to a priority of 1 is 2 hours, the recalculated value of due date is 2 hours added to the current date and time.

        • Note: If your system administrator has enabled the When the Priority changes, recalculate the Due Date based on the Current Date without user input check box in Application Settings, the Due Date Calculation window is not displayed and the due date is recalculated based on the current date.

        If your system administrator has enabled the Use the Organization's default business hours to calculate the Due Date check box in Application Settings, the due date is calculated by using the default business hours, time zone associated with the business hours, and holidays that are configured for your organization. For example, the system administrator has configured the default business hours as 10:00 a.m. to 6:00 p.m. for your organization. If you create a problem at 5:00 p.m. on a Monday and assign it a priority of 1, the due date of the problem is calculated as 11:00 a.m. of Tuesday, the next working day. Now, if the system administrator has configured Tuesday as a holiday, the due date of the problem is calculated as 11:00 a.m. of Wednesday, the next working day.
        Contact your system administrator to configure the business hours and holidays of your organization.

    7. Closed Date—This field is filled with the date and time when the problem is closed.

      8. Note: Some of the fields are filled with values when you execute a menu command or when associated data is retrieved.

  16. In the Broadcast section, enter the following information:
    1. Broadcast ID—Select the broadcast that is associated with the problem.
    2. Message—This field is filled with the broadcast message of the broadcast.
  17. Click Save.

    18. Note: When you open a new form to create a record and you save the record, you cannot navigate to the next or previous record by using the Next Record and Previous Record buttons in the upper-right section of the form. These buttons are disabled in a new form.

  18. After creating the problem, you can add supporting information for the problem.

    19. For more information, see Managing supporting information for problems.

  19. (Optional) Select the Inactive check box to designate the current record as inactive.

    20. You might need to make a record inactive because the record is no longer valid, but must be retained for compliance with the records retention policy.

    Note: Records marked as inactive are not included in any search performed on the database.

    The Supporting Information section is not loaded in a new form. When you save the record, the Show Supporting Information link appears. Click this link to view the Supporting Information section. You can configure the Supporting Information section to always display by selecting the Always show Supporting Information for existing records check box. For more information, see Managing supporting information for problems.

Managing supporting information for problems

The Supporting Information section at the lower section of the problem form allows you to:

    • Track the history of a problem
    • Link incidents, services, configuration items, and change requests
    • Add new change requests and tasks
    • Add documentation
    • View audit information about a problem

The problem form contains the following tabs in the Supporting Information section:

    • History—Allows you to track the history of all actions that have been performed on the problem by viewing the existing user-defined and system-defined actions or adding new user-defined actions.
    • Incidents—Allows you to link existing incidents that are the source of the problem.
    • Change Requests—Allows you to add new change requests and link existing change requests that must be used to resolve the problem. When a change request that is linked to a problem is closed, an email message can be sent to the problem owner (user) that the change request is closed. If the problem is assigned to a queue, no email notification is sent. To enable sending the email notification, contact your system administrators to activate the Notify problem owner when change request linked to problem is closed workflow.
    • Services and CIs—Allows you to link the services and configuration items that are affected when a problem is resolved.
    • Tasks—Allows you to add new tasks that must be performed to resolve a problem.
    • Documentation—Allows you to attach documents and URLs that are required to close a problem.
    • Auditing—Displays the history of changes that are made to fields in the form. The system administrator enables field history tracking for the required fields.
