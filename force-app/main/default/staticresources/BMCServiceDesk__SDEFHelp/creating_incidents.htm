When a client contacts the service desk with an issue, you first determine the nature of the issue. Before creating an incident, ensure if a similar incident is reported. If the client had reported an issue earlier, search the incident and update the client with the current status. If possible, you resolve and complete the incident. Otherwise, ensure that the incident is assigned to the appropriate profile and staff member. Create a new incident by capturing key information about the client and the issue.

Note: Create service requests when a client is requesting a service from the service desk. For example, request for a mobile phone or a new laptop. Create incidents when client reports an issue. For example, if the mobile phone or laptop of a client is not working or a client is having an issue when using the mobile phone or laptop.

BMC Remedyforce provides the AutoComplete feature for the First Name, Last Name, Client ID, Category, Impact, and Urgency fields in this form.

To create an incident in Visualforce page layout UI

  1. On the BMC Remedyforce home page, click Workspaces.
  2. From the list of Workspaces, expand Incident Management.
  3. In the Incident Management list, click Incidents.
  4. In the incidents list view, click New.

    5. Note: You can view the Incident Management process of the Alignability Process Model by clicking Incident Management Process. For more information, see Incidents list view Views menu.

  5. Click Template.

    6. To help ensure accuracy and completeness, BMC recommends that you use a template to create an incident record. A template ensures consistency in the way information about an incident is captured. Your administrator can define templates for commonly occurring incidents.

  6. In the Select From Templates window, double-click the required template.

    7. The values of the fields in the template are filled in the corresponding fields of the Incident form.

  7. Click the Last Name list.

    8. Depending on how system administrator has configured the settings, the Select From Clients or Search Clients window is displayed. If your system administrator has configured Application Settings to allow you to create incidents for contacts and leads, the Search Clients window is displayed where you can search for a user, contact, or lead. A list of matching entries is displayed under separate tabs of Users, Contacts, and Leads.

    If your system administrator has configured Application Settings to prevent you from creating incidents on behalf of contacts and leads, the Select From Clients window is displayed with a list of users. Double-click a record to select it.

    The type of client, user, contact, or lead, is displayed in the incidents list view in the Client Type and Client Name columns. The Client ID column in the incidents list view is blank if the incident record is created for a lead or a contact.

  8. In the Search Clients window, in the Keywords field, enter the search keywords and press Enter or click .

    9. The list of matching records is categorized under Users, Contacts, and Leads (if applicable). You can change the number of records displayed in the list in the Maximum Records field. A maximum of 200 records per object can be displayed.

    The data fields, such as account ID, Mobile Phone, Status, and so on that display data of the matching records, can be configured. Contact your system administrator to configure the fields that you want to view in the search results.

  9. To select a record, double-click the appropriate record.

    10. The First Name, Client ID, and Account fields are filled with the first name, client ID, and account (if exists) of the selected client. You can change the account of the client.

    The icon displays the same fields that are displayed in the Search Clients window, such as Account ID, Mobile Phone, and so on of the selected record in a pop-up window. Contact details are the mandatory fields that are displayed. The fields that are displayed in the pop-up window can be configured on the Global Search Settings page. Contact your system administrator to configure the fields for your organization. This icon is displayed on custom incident form.

    Users with profiles that do not have access to the lead object, such as Standard Platform User, cannot view the client details pop-up window, if lead is selected in the Client ID field on the custom Incident form.

  10. To change the account, click the Account list.

    11. In some cases, you might have to create an incident for the parent account of the client's account when the applicable Service Level Agreement (SLA) exists with the parent account. For example, when the client's account is Pre-Sales and you have an SLA with the parent account of the Pre-Sales account (Sales account), you select the Sales account in the Account list.

  11. Click the Category list.
  12. In the Select From Categories window, perform the following actions:
    1. Select the category of the incident.

      2. Categories allow you to classify the incident by using standard classifications, which can be easily tracked for reporting purposes.

      The Select From Categories window displays incident categories and categories that are available to both Incident Management and Service Request Management. Service Request categories appear in this window if they are parent categories to Incident categories. If you want to select a Service Request category for the incident, select Filter Options > Service Request Categories.

      If you click the Category list after selecting a service request definition in the Request Definition list, the Select From Categories window displays Service Request categories and categories that are available to both Incident Management and Service Request Management. Incident categories appear in this window if they are parent categories to Service Request categories. If you want to select an incident category for the service request, select Filter Options > Incident Categories.

    2. To use the standard description of the selected category, select the Standard Description check box.

      3. The standard description of the selected category is appended to the Description field of the incident form.

    3. To use the standard resolution of the selected category, select the Standard Resolution check box.

      4. The standard resolution of the selected category is appended to the Resolution field of the incident form.

    4. Click OK.

      5. You select a value in the Request Definition list when you are creating a service request. For more information, see Creating a service request in Visualforce page layout UI.

  13. In the Description field, type the symptoms reported by the client.

    14. The incident description is stored as part of this incident and becomes part of your database for solving future incidents. Check the text for accuracy and spelling to ensure that you retrieve right results when searching the database later.

  14. In the Resolution field, type the resolution that must be implemented for this incident.
  15. If this incident is resolved with the first telephone call made to the staff member, select the First Call Resolution check box.
  16. In the Note field, type notes about the incident.

    17. When customers call the service desk to provide updates about their incidents, the staff member can type these updates in the Note field, and click the Add icon. These updates are added to the Supporting Information section as history records in the History tab.

    The Note field allows staff members to add notes about the incident without having to navigate to the Supporting Information section, and create a history record.

    If the text in the Note field is long, first few words are displayed in the Note column in the Supporting Information section and the remaining content is displayed in a ToolTip. If text in the Note field is more than 500 characters, first 500 characters are displayed in the tooltip and ellipses are displayed in the end. To view complete text, double-click the record and the text is displayed in the Note field.

    Note: The Note and Duration fields are enabled only after you save the record.

  17. In the Duration field, type the amount of time that you spent working on the incident.
  18. Click Add.

    19. The Supporting Information section is displayed. The contents of the Note and Duration fields are added to the Supporting Information section as a history record in the History tab.

  19. In the Details section, enter the following information:
    1. Status—Select the status that you want to assign to the current incident.

      2. You can track the progress of an incident from its stages of opening to closure by using this field.

      The Incident Progression Bar in the header of the incident form is updated depending on the stage that is linked to the status that you have selected. For example, if the Working status is linked to the In Process stage and you select Working as the status of the incident, the In Process stage is selected in the Incident Progression Bar.

      Note: You can assign an incident to the logged-on user by changing the status of the incident to a status with the Acknowledged stage. Contact your system administrator to enable the Assign the Incident to the logged in user when a status with the Acknowledged stage is selected check box in Application Settings to allow this quick assignment.

    2. Impact—Select the impact that you want to assign to the current incident.

      3. The level of impact and the urgency are used to calculate the priority of the incident.

    3. Urgency—Select the urgency you want to assign to the current incident.

      4. The urgency and the level of impact are used to calculate the priority of the incident.

    4. Priority—When you select an urgency and impact and save the record, the Priority field is filled in. The priority is calculated depending on the urgency and impact values that you have selected.
    5. Opened Date—This field is filled with the date and time at which the incident is created.
    6. Responded Date—This field is filled with the date and time when you select Actions > Responded.
    7. Due Date—When you select an urgency and impact and save the record, the Due Date field is filled in.

      8. The due date is calculated based on the duration of the priority. If required, you can select a different due date from the calculated due date.

      The due date of the incident changes when qualifying service targets with Resolution as the Target Type are available, and these service targets are applied after saving the incident. When a service target with Resolution as the Target Type is applied, the Due Date field is disabled. The Due Date Calculation window does not appear when the Due Date field is disabled. The due date of the incident goes through further updates based on changes to the service targets.

      Note: The Due Date field is not disabled if the applied service targets of the incident are either in the Not Started or Canceled state.

      If the service targets of the incident are attached and in the State of Canceled, the Due Date field is enabled after the saving the incident and the Due Date Calculation window appears when you change the priority of the incident. For more information, see Recalculating the Due Date with service targets.

      When you are editing an incident and you update the impact or urgency that leads to a change in the priority of the incident, the Due Date Calculation window appears. The following options are available in the Due Date Calculation window:

      • Do not Recalculate—The existing due date is retained for the incident.
      • Recalculate Based on Open Date—The due date is recalculated by using the opened date. For example, if the priority is changed to 1 and the duration assigned to a priority of 1 is 2 hours, the recalculated value of the due date is 2 hours added to the date and time when the incident was created.
      • Recalculate Based on Current Date—The due date is recalculated by using the current date. For example, if the priority is changed to 1 and the duration assigned to a priority of 1 is 2 hours, the recalculated value of the due date is 2 hours added to the current date and time.

        • Note: If your system administrator has enabled the When the Priority changes, recalculate the Due Date without user input check box in Application Settings, the Due Date Calculation window is not displayed and the due date is recalculated based on the Use Opened Date instead of Current Date to recalculate Due Date setting.

        Note: If your system administrator has enabled the Use the Organization's default business hours to calculate the Due Date check box in Application Settings, the due date is calculated by using the default business hours, time zone associated with the business hours, and holidays that are configured for your organization. For example, the system administrator has configured the default business hours as 10:00 a.m. to 6:00 p.m. for your organization. If you create an incident at 5:00 p.m. on a Monday and assign it a priority of 1, the due date of the incident is calculated as 11:00 a.m. of Tuesday, the next working day. Now, if the system administrator has configured Tuesday as a holiday, the due date of the incident is calculated as 11:00 a.m. of Wednesday, the next working day.
        Contact your system administrator to configure the business hours of your organization (by navigating to Setup > Company Profile > Business Hours). Contact your system administrator to configure holidays for your organization (by navigating to Setup > Company Profile > Holiday).

    8. Closed Date—This field is filled with the date and time at which the incident is closed.
    9. User Called—This field is incremented by 1 when you select Actions > Increment Call Counter.

      10. This field allows you to track the number of times a client of an incident has called the service desk.

    10. Problem #—Click this button to open the problem form where you can create a problem record that is added to this incident. The values of the Category, Description, Resolution, Impact, Urgency, and Priority fields of the incident record are populated in the problem form. When you save the problem record, the problem form remains open, and you are taken to the incident form where the problem # of the problem record is populated in this field. If you are creating a new incident, you must save the incident record to link the problem to the incident. If you are editing an incident, the problem is added to the incident when you save the problem record in the problem form.

      11. Note: Some fields are filled with values. This might occur when you execute a menu command or when associated data is retrieved. For example, when you execute the Actions > Responded menu command, the Responded field is filled with the current date and time, which allows you to track the date and time when you have responded to the client.

Recalculating the Due Date with service targets

If a qualifying incident service target with Resolution as the Target Type exists, BMC Remedyforce replaces the Due Date of the incident with the Target End Date of the service target. If the Due Date of the incident is not populated, BMC Remedyforce populates the Target End Date to the Due Date of the incident. If a service target is cancelled or modified and a new qualifying service target is available, BMC Remedyforce applies the new service target to the incident and replaces the Due Date of the incident with the Target End Date of the new service target.

When multiple service targets qualify for the incident, BMC Remedyforce populates the nearest Target End Date in the Due Date of the incident. For example, if you have two qualifying service targets with a Target End Date of 9/25/2011 10:30 p.m. and a Target End Date of 9/26/2011 10:30 p.m., BMC Remedyforce populates the Due Date with the Target End Date of 9/25/2011 10:30 p.m.

If the clock of the service target is paused for some time after the service target is applied, the duration for which the clock is paused is added to Target End Date and this updated value is populated in the Due Date of the incident.

Supporting information for incidents and service requests

The Supporting Information section at the lower section of the incident form allows you to:

    • Track the history of the incident
    • Add new tasks and change requests
    • Link additional CIs
    • Link existing change requests
    • Add documentation
    • View audit information about the incident

The incident form contains the following tabs in the Supporting Information section:

    • History—Allows you to track the history of all actions that have been performed on the incident by viewing the existing user-defined and system-defined actions or adding new user-defined actions.
    • Tasks—Allows you to add new tasks that must be performed to resolve the incident. BMC Remedyforce displays inactive tasks in the Supporting Information section if your system administrator has configured the settings.
    • Additional CIs- Allows you to link additional CIs to the incident.
    • Change Requests—Allows you to add new change requests and link existing change requests that must be used to resolve the incident.
    • Documentation—Allows you to attach documents and URLs to the attachments that are required to close the incident.
    • Auditing—Displays the history of changes made to fields in the form. The system administrator enables field history tracking for required fields.
