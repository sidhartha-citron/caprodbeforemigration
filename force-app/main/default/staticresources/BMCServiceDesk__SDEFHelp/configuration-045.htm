BMC recommends that you implement the following best practices when you customize forms:

Situation
Description
Adding fields to a custom form
BMC recommends that if you add a field to a custom form by using one field set, you must avoid adding the same field to the custom form through another field set.
Main Panel: Second Column field set
This field set is hidden and does not appear if you do not add a field from this field set. This field set appears as a second column and it is parallel to the Main Panel: First Column field set. BMC recommends that you use this field set to add more fields that you want staff members to access easily rather than staff members having to navigate to the Details section or Additional Details section to access these fields. Horizontal and vertical scroll bars might appear on the custom form when you collapse the Navigator bar and Sidebar.
No fields in the In the Field Set bucket of a field set
If you have removed all fields from the In the Field Set bucket, the field set does not appear on the custom form. For example, if you have removed all fields from the In the Field Set bucket of the Additional Details field set, the Additional Details section does not appear on the custom form. BMC recommends that you remove all fields from the In the Field Set bucket to remove a field set from the custom form.
Text Area (Rich) custom fields
If you create a Text Area (Rich) custom field and add it to a custom form, this custom field appears as a Text Area (Long) field on the custom form with a maximum limit of 32000 characters and without any Rich Text abilities. BMC recommends that you avoid using Text Area (Rich) custom fields in field sets.
Picklist (Multi-select) custom fields
If you create a Picklist (Multi-select) custom field and add it to a custom form, this custom field appears distorted to users of those profiles that have the Read-Only field-level security for this field. BMC recommends that you grant the Visible field-level security to all profiles to prevent this field from distorting on a custom form. For more information about granting field-level security, see https://na11.salesforce.com/help/doc/user_ed.jsp?loc=help&target=adding_fields.htm&section=Customizing and https://na11.salesforce.com/help/doc/user_ed.jsp?section=help&target=admin_fls.htm&loc=help&hash=topic-title.
Referenced fields in custom forms
Custom forms do not display referenced fields, that is, fields which are linked by a Lookup field. To use reference fields in the custom form, you must create a formula field to display the values of the referenced fields. For more information about creating Formula fields, see https://na11.salesforce.com/help/doc/user_ed.jsp?section=help&target=customize_formulas.htm&loc=help&hash=topic-title.
For example, in the CI Detail section, the Incident custom form displays only the Configuration Items field. However, the Incident standard form displays the Configuration Items field and its referenced fields, such as Asset #, Serial #, and Warranty Expiration Date fields in the CI Detail section. When you select a CI in the Configuration Items field, the associated Asset #, Serial #, and Warranty Expiration Date values are populated in the Incident standard form. To use reference fields in the Incident custom form, you must create a Formula field to display the values of the referenced fields, such as Asset #, Serial #, and Warranty Expiration Date fields, in the CI Detail section.
Mandatory fields in field sets
If you create a mandatory custom field for a custom object and you use this custom field in a field set, you must configure a default value for this custom field. If you do not provide a default value, you cannot create records for the custom object.
If you add mandatory custom fields with default values to a custom object and you do not add them to the custom form, you cannot update and save existing records in the custom form. You receive the following error message: 
Required fields are missing: [<mandatory custom 
field names>]
BMC recommends that you add the mandatory custom fields to the custom form before you update existing records.
Custom fields and the Additional Details field set
By default, there are no fields in the In the Field Set bucket of the Additional Details field set, and this field set does not appear on the custom form. If you create custom fields for a custom object, these custom fields are added to all field sets of the custom object. BMC recommends that you add these custom fields to the custom form from the Additional Details field set. This allows you to manage the custom fields that you have created for the custom object. For more information about creating custom fields, see https://na11.salesforce.com/help/doc/user_ed.jsp?loc=help&target=adding_fields.htm&section=Customizing.
Using templates to create records in custom forms
If you have a template that contains values for all fields in a field set and you use the template to create records, the template populates values only for those fields that are visible on the custom form. For example, if you assign a value for the Configuration Items field in an Incident template and use this Incident template to create an incident record, the value of the Configuration Items field is not populated in the incident record because you have not added the Configuration Items field to the Incident custom form. BMC recommends that if you are using templates, you must add all fields in the template to the custom form before you create records from these templates.
In a custom form, if you use a template with fields that are not added to the custom form and these fields are populated with values, the record is created successfully, but these template fields are populated in the record. Later on, if you add these template fields to the custom form, the values from these fields are displayed when you view the record. For example, you assign a value for the Configuration Items field in an Incident template and this field is not added to the Incident custom form. When you use this Incident template to create an incident record, the value of the Configuration Items field is not populated in the incident record because you have not added the Configuration Items field to the Incident custom form. When you add the Configuration Items field to the Incident custom form, the value from the template is populated in the incident record.
Deleting custom fields in a field set
If you create a custom field, add it to a field set, then later delete the custom field, staff members must refresh their browsers to prevent the following error message from appearing: 
The configuration of your org has changed, 
please reload the page. Missing dependent 
object: Field: <deleted custom field name>
Custom fields of the Lookup Relationship data type for users without the Read permission on the object
If you have created a custom field of the Lookup Relationship data type and users do not have the Read permission on the object to which the custom field is related to, this custom field does not appear in the custom forms for users without the Read permission when the custom field is added to the field set of the custom object.
For example, users of the Change Manager profile do not have the Read permission on the Lead or Case objects. In Incident object, if you create custom fields of the Lookup Relationship data type on the Lead or Case objects and add the custom fields to any of the field sets of Incident object, these custom fields are not displayed on Incident custom form for users of the Change Manager profile.
