Work Instructions for Change Coordinators

6.8.1

After you have been notified that the work on the change has been completed, change the Status field of your task for updating the requesters of the change. If you are not ready to work on the task, change the value from Registered to Accepted. When you are ready to update the requesters of the change, set the value to In Progress.

6.8.2

If the change implementation was not successful (for example, because the change request was rejected or the change implementation had to be rolled back), specify the reason in the Notes action of the change.

6.8.3

Send an email to the approvers of the change telling them whether the change was implemented successfully. If the change implementation was not successful, include the reason for the failure.

6.8.4

From the Incidents tab of the change, open each change request that was received from Incident Management as an incident (if any). If the change implementation met the change requirements specified in the incident, summarize how the change was implemented in the Resolution field of the incident. If the change implementation did not meet the change requirements specified in the incident, specify the reason for the failure in the Resolution field of the incident.

6.8.5

From the Problems tab of the change, open each change request that was received from Problem Management as a problem (if any). If the change implementation met the change requirements specified in the problem, summarize how the change was implemented by adding a Notes action to the change,  and set its Status field to Completed. If the change implementation did not meet the
change requirements specified in the problem, specify the reason for the failure by adding a Notes action to the change. Set the Status field of such problems to Rejected if the change was rejected by its approvers or to Completed if the implementation of the change did not fulfill the requirements specified in a problem (for example, because the implementation was only partially successful or had to be rolled back).

Note:

Setting the Status field of a problem to Rejected or Completed causes a problem to be returned to its problem coordinator.

6.8.6

If the requirements for the change were received from Release Management as a document, send an email to the release coordinator to inform them whether the change was implemented successfully. If the change was not implemented successfully, include the reason for the failure in the email.

6.8.7

If the change implementation was performed to return a service to production after it had been recovered by Continuity Management, send an email to the service owner to inform them whether the change was implemented successfully. If the change was not implemented successfully, include the reason for the failure in the email.

6.8.8

When you complete the update of the requesters, set the Status field of the task to Completed.

6.8.9

In the Duration (HH:MM) field, enter the time that you spent completing the task.
