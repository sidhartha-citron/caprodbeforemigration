Work Instructions for Specialists

5.1.1

When the Status field of your task for the development of the new release changes from Registered to Assigned, open the task and read the requirements in the Description field.

5.1.2

If you are not ready to start working on the task, set the Status field of the task to Accepted.

Note:

If the task should be assigned to another developer, ask the change coordinator to change its assignment.
To do so, set the Status field of the task to Rejected and add a Notes action saying why you rejected the task.

5.1.3

When you are ready to start developing the new release, set the Status field of the task to In Progress and develop the new release.

Note:

If you cannot continue your work on the task (for example, the development is more difficult than anticipated and you are waiting for help from a colleague), set its Status field to Waiting for, and add in a Notes action saying what you are waiting for and when you expect to complete the task.

5.1.4

When you complete the development of the new release, test it to ensure that the change requirements have been met. Do this in a separate test environment if one is available. Otherwise, perform your tests in the development environment. If (automated) scripts for regression testing are available for the application, update and run them. Use the development environment to fix any errors that occurred during your tests. After you have confirmed that the change requirements have been met and no new issues have been introduced, specify which tests you have performed in the Result field of the task and set its Status field to Completed.

If the requirements of the change could not be met, set the Status field of the task to Failed and specify in the Result field why it was not possible to develop the required functionality.

5.1.5

In the Duration (HH:MM) field, enter the time that you spent completing the task.
