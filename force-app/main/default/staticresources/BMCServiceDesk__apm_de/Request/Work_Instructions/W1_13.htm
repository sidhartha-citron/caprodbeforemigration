Arbeitsanweisungen für Service Desk Analysts

1.13.1

Wählen Sie die entsprechende Anforderungskategorie im Feld Kategorie aus.

1.13.2

Wenn Sie eine Serviceanforderung erstellen, wählen Sie die
Anforderungsdefinition, die im Feld Anforderungsdefinition verwendet werden soll. Die Anforderungsdefinitionen für
die gewählte Kategorie werden angezeigt.

1.13.3

Geben Sie eine kurze, aussagekräftige Beschreibung der Anforderung im Feld Beschreibung ein. .

1.13.4

Wählen Sie die Art der Übermittlung an den Service Desk im Feld Incident Source (Incident-Quelle) aus. .

1.13.5

Wählen Sie den verknüpften Service im Feld Service.

Hinweis:

Falls die Anforderung eine Beschwerde, ein Lob oder eine Anforderung zur Entwicklung eines neuen Service beinhaltet, wählen Sie den Service N/A - Service does not (yet) exist or is out of scope (N/A – Service ist (noch) nicht vorhanden oder außerhalb des Anwendungsbereichs) aus.

1.13.6

Wenn sich die Anforderung auf ein bestimmtes Konfigurationselement (CI) bezieht (oder vermutet wird, dass ein bestimmtes CI sie verursacht hat), wählen Sie das CI im Feld CI aus.

Hinweis:

Falls das CI nicht in der CMDB registriert ist, jedoch in den Bereich des Konfigurationsmanagement-Prozesses fällt, wählen Sie das CI mit dem Code NORECORD aus.

Hinweis:

Falls die Anforderung kein bestimmtes CI betrifft oder das CI nicht unter den Konfigurationsmanagement-Prozess fällt, wählen Sie das CI mit dem Code N/A aus.

1.13.7

Speichern Sie die Anforderung.

1.13.8

Falls der Anforderungstyp auf „Serviceanforderung“ gesetzt ist, werden die dazugehörigen Arbeitsaufträge generiert und es wird ein Popup eingeblendet, in dem Sie die Fragen zur Serviceanforderung beantworten müssen.

1.13.9

Fügen Sie eine Hinweisaktion hinzu, in der Sie die zusätzlichen Informationen angeben, die der Benutzer bereitstellen kann und die bei der Lösung der Anforderung hilfreich sein können.

1.13.10

Falls der Anforderungstyp auf „Incident“ gesetzt ist, prüfen Sie das Service Management, um festzustellen, welche der Changes, die innerhalb der letzten 24 Stunden implementiert wurden, zum Incident geführt haben könnten. Falls Sie einen Change identifiziert haben, der den Incident verursacht haben könnte, verknüpfen Sie den Change in der zugehörigen Liste Verknüpfte Change Requests. Fügen Sie eine Hinweisaktion hinzu, in der Sie angeben, weshalb der Change zum Incident geführt hat.
