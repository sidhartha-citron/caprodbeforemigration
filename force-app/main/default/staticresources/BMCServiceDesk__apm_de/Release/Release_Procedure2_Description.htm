Verfahren 2, Release-Definition


Wenn die nächste CAB-Besprechung für einen Service ansteht, sendet der Release-Koordinator des Service die Einladungen an die CAB-Mitarbeiter des Service. In der Einladung gibt der Release-Koordinator die Deadline an, bis zu der Change Requests zur Prüfung bei der nächsten CAB-Besprechung gesendet werden können. Nachdem die Deadline für das Senden von Change Requests abgelaufen ist, stellt der Release-Koordinator die Liste mit Change Requests, die bei der nächsten CAB-Besprechung geprüft werden sollen, fertig und sendet sie an die CAB-Mitarbeiter, sodass diese die Change Requests vor der Besprechung intern diskutieren können.

Die fertige Liste mit Change Requests wird während der Besprechung geprüft. Nach Prüfung aller Änderungen teilt jeder CAB-Mitarbeiter dem Release-Koordinator mit, welche Change Requests abgelehnt werden müssen und welche im nächsten Release enthalten sein sollen. Ein Change Request wird abgelehnt, wenn mindestens zwei Drittel der CAB-Mitarbeiter der Meinung sind, dass er abgelehnt werden sollte. Ein Change Request ist dagegen im nächsten Release enthalten, wenn mindestens zwei Drittel der CAB-Mitarbeiter darin übereinstimmen, dass er enthalten sein sollte. Change Requests, die weder abgelehnt noch im nächsten Release enthalten sind, werden bei der nächsten CAB-Besprechung erneut geprüft. Nach Abschluss der CAB-Besprechung bestimmen die CAB-Mitarbeiter Datum, Uhrzeit und Ort der nächsten CAB-Besprechung.

Nach der CAB-Besprechung stellt der Release-Koordinator sicher, dass die Anforderer des abgelehnten Change Requests informiert werden, indem er diese Change Requests abschließt. Anschließend veröffentlicht der Release-Koordinator das Protokoll der CAB-Besprechung und registriert den neuen Release. Der Release-Koordinator verknüpft einen Change für die Koordination des Release und verknüpft die Change Requests, die im Release enthalten sein sollen, mit diesem Change.

Der Release-Koordinator überprüft dann diese Change Requests zusammen mit dem Change-Koordinator des Service, für den der Release implementiert werden soll, und den Change-Koordinatoren der unterstützenden Services, die ebenfalls geändert werden müssen. Sie teilen die Anforderungen der verschiedenen Change Requests untereinander auf und entwerfen anschließend einen umfassenden Implementierungsplan, der die Dauer jedes Change sowie die Abhängigkeiten dieser Changes untereinander angibt.
