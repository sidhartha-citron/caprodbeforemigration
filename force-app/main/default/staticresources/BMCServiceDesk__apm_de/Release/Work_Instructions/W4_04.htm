Arbeitsanweisungen für Release-Koordinatoren

4.4.1

Besprechen Sie die Situation mit den Koordinatoren der im Release enthaltenen Changes, um zu ermitteln, wie sich der (partielle) Fehlschlag der Change-Implementierung auf die gesamte Implementierung des Release auswirkt.

4.4.2

Einigen Sie sich mit den Change-Koordinatoren auf die bestmöglichen Maßnahmen nach dem Fehlschlag. Es könnte entschieden werden, je nach Bedarf zusätzliche Changes zu implementieren und damit sicherzustellen, dass alle bzw. die meisten Anforderungen weiterhin erfüllt werden können. Wenn hingegen die Kosten und der Aufwand für die Implementierung (eines Teils) der unerfüllten Anforderungen zu hoch oder unbekannt sind, könnte entschieden werden, einen vollständigen Rollback des Release durchzuführen. In einem solchen Fall müssen die nicht implementierten Change Requests erneut geprüft werden, um zu einer realistischeren Kosten- und Aufwandschätzung zu gelangen.

4.4.3

Falls entschieden wurde, einen oder mehrere verbleibende offene Changes für einen Release nicht zu implementieren, setzen Sie die zuständigen Koordinatoren davon in Kenntnis, dass Sie die Anforderung für diese Changes zurückgezogen haben. Prüfen Sie anschließend, ob für die Fertigstellung dieser Changes das Feld Abschlusskategorie auf Request Withdrawn by Requester (Anforderung durch Anforderer zurückgezogen) gesetzt ist.
