Work Instructions for Release Coordinators

4.9.1

Add a Notes action in the release to summarize which requirements of which change requests have not been fulfilled (if any).

4.9.2

Open the change request to which the change requests for the release are linked. From this change request, open the task for updating the requesters and approvers of the release. This task is generated by the change template when the change is registered.

4.9.3

Update the Due Date field of the task by specifying a date and time that can be achieved.

4.9.4

Ensure that you are selected in the Owner field of the task.

4.9.5

If you are not ready to perform the requester and approver update, change the Status field of the task from Registered to Accepted. When you are ready, set the Status field of the task to In Progress.

4.9.6

In the Linked Incidents section of the change request to which change requests for the release are linked (this is also the change request to which the task for updating the requesters and approvers of the release is linked), open each request whose requirements are fully met by the implementation of the release. In the Resolution field of such requests, summarize how the release was implemented, and set the Closure Category field to Solved - Root Cause Analysis not Required.

In the Duration (HH:MM) field, in the Notes action, enter the time that you spent updating each request.

4.9.7

Set the Status field of the request to Completed to ensure that the service desk informs the client who is linked to the request in the Client ID field.

4.9.8

In the Linked Problems section of the change request to which change requests for the release are linked (this is also the change request to which the task for updating the requesters and approvers of the release is linked), open each problem whose requirements are fully met by the implementation of the release. In the Action History section of such problems, summarize how the release was implemented, and set the Status field to Change Completed.

In the Duration (HH:MM) field, in the Notes action, enter the time that you spent updating each problem.

Note:

Setting the Status field of a problem to Change Completed causes a problem to be returned to its problem coordinator.

4.9.9

Open each request whose requirements are not (fully) met and estimate the effort of internal employees and long-term contractors that is still needed to fully implement the requirements. Also, estimate the cost of CIs, services and support that still need to be acquired from suppliers for each of these support requests to ensure that their requirements are fully met. Add a Notes action for each of these requests to specify:

  • Which of their requirements were not met and why that is the case

  • The new cost and effort estimates and the assumptions on which they are based

  • That the unfulfilled requirements will be reviewed again during the next CAB meeting of the service

Relate each of these requests to the change that acts as a placeholder for requests that are waiting to be reviewed during the next CAB meeting of the service, and unlink those requests from the change that is linked to the release. Also, send an email message to the client who is selected in the Client ID field of each of these requests. In each email, include the information that you entered in the Notes actions of the email recipient's request.

In the Duration (HH:MM) field, in the Notes action, enter the time that you spent updating each request.

4.9.10

Open each problem whose requirements were not (fully) met by the implementation of the release. Add a Notes action in such problems to specify why the requirements were not met. Set the Status field of such problems to Change Completed to ensure that the problems are returned to the problem coordinators. Unlink such problems from the change request to which the change requests for the release are linked.

In the Duration (HH:MM) field, in the Notes action, enter the time that you spent updating each request.

4.9.11

Prepare an email for the CAB staff members (if the release was defined by the CAB) and the approvers of the business case (if one had to be prepared for the release). Specify in the email whether or not all requirements have been met. If all requirements are not met, specify those requirements and give reason why it was decided to complete the release without fulfilling these requirements.

4.9.12

Set the Status field of the task for updating the requesters and approvers of the release to Completed.

4.9.13

In the Duration (HH:MM) field, in the Notes action, enter the time that you spent updating the requesters and approvers.

4.9.14

Select the appropriate option in the Closure Category field of the release. Set the Status field of the release to Completed.
