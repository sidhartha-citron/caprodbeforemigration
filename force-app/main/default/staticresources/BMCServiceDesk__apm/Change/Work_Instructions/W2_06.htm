Work Instructions for Change Coordinators

2.6.1

If the change causes one or more customer organizations to receive different functionality or to experience a period of service degradation or service outage (regardless of whether it occurs during service hours), add a Notes action in the implementation tasks which execution is going to affect them and specify these organizations.

2.6.2

Link each implementation task to the CIs that change (if any) when the task is executed by clicking New Link in the Linked Services and CIs section of the task.

2.6.2

Add a Notes action to indicate the expected downtime for each implementation task.

2.6.3

For each implementation task, determine who is the most appropriate staff of your group (in terms of skills, availability, and access rights) to execute it. Select this staff in the Owner field of the task.

If an implementation task must be completed by another group, work with the coordinator of the group to determine who is the most appropriate staff (in terms of skills, availability, and access rights) to do the work and agree on a realistic completion target. Select that staff in the Owner field of the task. However, if the group coordinator indicates that it is not known who can complete the task, select the group in the Owner field of the task. In that case, the group coordinator can select the most appropriate staff in the Owner field after the status of the task has been set to Assigned.

If an implementation task must be completed by an external organization, work with your contact person at this organization to determine a realistic implementation target. Select yourself in the Owner field of the task and add a Notes action to enter the external organization, which is asked to do the work. Later, when the work starts, inform the supplier end enter the reference number, under which the supplier has registered this assignment, in the Notes action.

2.6.4

Ensure that the date and time specified in the Due Date field of the tasks is achievable. For non-standard or standard changes that require risk and impact analysis, register the start date of the first implementation task in the Scheduled Start field and the target date of the last implementation task in the Scheduled End field.

2.6.5

Set the Status field of the implementation task to Registered.

2.6.6

In the Back-out Plan field of the change request, specify how the change can be rolled back if necessary. For example, If a rollback is required, restore the backup that is taken before the new release is transferred to production.
