役割と責任


次の表に、プロセスモデルに定義されているさまざまなプロセスに関係する役割と、各プロセスでその役割が持つ責任を示します。プロファイルをレビューするには、役割をクリックします。

役割

責任

CABメンバー

変更管理

変更管理者からリクエストを受けた場合、変更とその実装について承認または却下します。

リリース管理

CABレビュー用に収集された変更リクエストのうち、却下するもの、次にリリースするもの、CABの次のミーティングで再度レビューするものを決定します。

変更コーディネータ

変更管理

  • 作成元がリクエスト管理、問題管理、リリース管理、継続管理である変更リクエストを評価します。

  • 変更リクエストを処理するために、必要に応じて変更を作成します。

  • リスクとインパクトの作業依頼を作成して割り当て、リスクとインパクト分析の調整を行います。

  • 可能な場合はリスクとインパクトの作業依頼に回答します。

  • 実装作業依頼を作成して割り当て、実装計画を作成します。

  • 変更の進捗状況を監視します。

  • 変更を終了します。

変更管理者

変更管理

  • リスクとインパクト分析を見直して、これらが完全に実行されていることを確認します。

  • 変更の実行中に発生する障害のリスクと顧客への影響の両方を最小限に抑えるために、適切なアクションが計画されていることを確認します。

  • 計画された実行の時期が、他の計画された変更またはイベントと重ならないことを確認します。

  • 承認作業依頼を作成して適切なサービスオーナーおよび顧客担当者に割り当て、変更の承認を取ります。

構成管理者

構成管理

  • 注文が必要な構成アイテム（CI）の購入のために、購入申請を送信します。

  • 納品後は、構成管理者のグループの担当になったCIに、ラベルを貼付するようにします。

  • 構成管理者のグループが担当するCIの情報を、最新の状態に保ちます。

  • 構成管理者のグループが担当するCIを、提供またはサポートした社外組織の情報を保守します。

  • 構成管理者のグループが担当するCIのサポート連絡先を、登録および更新します。

  • 構成管理者のグループが担当するCIのリンクを保守します。このリンクとは、CIと他のCI、CIのサプライヤ、CIのサポート契約およびCIの一部であるサービスインスタンスとのリンクです。

顧客担当者

変更管理

アプリケーションの新しいリリースが、テスト環境に移行後、テスト済みであることを確認します。

グループコーディネータ

リクエスト管理

グループに割り当てられたサポートリクエストを、各スタッフのスキル、空き状況、アクセス権限の面で、適切なスタッフに割り当てます。

問題コーディネータ

問題管理

  • 問題を特定するため、自身が問題コーディネータであるサービスに関連するサービスリクエストをレビューします。

  • 可用性と容量管理プロセスで特定された問題を含め、担当する問題が、問題管理プロセスに従い、スケジュールどおりに、指定された優先順で処理されていることを確認します。

  • 問題で入力されている情報が正確で漏れがないことを確認します。

  • 構造的な解決策が見つかっていない問題に関して、定期的にレビューを行います。

  • 構造的な解決策を確認し、問題を終了します。

リリース管理者

変更管理

  • 開発環境で開発済みのアプリケーションの新しいリリースを、テスト環境に移行します。

  • テスト環境でテスト済みのアプリケーションの新しいリリースを、本番稼働環境に移行します。

リリースコーディネータ

リリース管理

  • リリースコーディネータとして処理を行うサービスの変更リクエストについて、変更管理から渡された後で確認します。

  • リリースコーディネータである担当者が行うサービスについてCABミーティングを準備し、円滑に進めます。

  • リリースの要件を論理グループに分類し、変更コーディネータが効率的に処理できるようにします。

  • 新しいリリースの実装に追加資金が必要な場合は、そのリリースに対するビジネスケースを作成します。

  • リリースの実装を開始し、必要に応じて修正アクションを決定します。

  • 実装後のミーティングを準備および開催し、今後のリリースに対する改善案を収集します。

サービスデスクアナリスト

リクエスト管理

  • サービスプロバイダ組織によって提供されたサービスのユーザー向けに、単一の連絡窓口を用意するように支援します。

  • ユーザーから必要な情報を得て、効率的に、正確に、完全にサポートリクエストに登録します。

  • アクセス権に関する制限および時間的制約の範囲内で、受領済みリクエストのうち、可能な限り多くを解決します。　

  • サービスデスクで解決できないリクエストは、解決するために、適切なグループに割り当てます。

  • 影響力の大きいインシデント（つまり、複数のユーザーに影響するサービス停止）の進捗状況を監視し、すべての該当するユーザーに通知します。

  • インシデントが関連SLAに指示された解決時間内に解決されない場合、影響力の大きいインシデントを影響するサービスのサービスオーナーにエスカレートします。

  • サポートリクエストが解決された後に、解決策をユーザーに通知して、解決策の確認を依頼します。

サービスデスクマネージャ

リクエスト管理

  • ユーザーには丁重に対応します。

  • 苦情と称賛カテゴリのサポートリクエストをすべて処理し、苦情を報告したユーザーが、サービスプロバイダ組織に満足するようにします。

サービスレベル管理者

サービスレベル管理

  • 変更管理からの要求に応じ、カタログ項目の追加と削除を行って、サービスプロバイダ組織のWebベースのサービスカタログで管理します。

  • 変更管理からの要求に応じ、SLAを用意し、署名してもらいます。

  • サービス管理アプリケーションでサービスとSLAを登録し、その情報を、その他のサービスプロバイダ組織に公開します。

  • 新しい顧客がサービスプロバイダ組織のサービスに加入するときに、個々の顧客、顧客組織、顧客担当者の連絡先の詳細を登録します。

  • 更新された連絡先情報を受信した後、個々の顧客、顧客組織、顧客担当者の情報を更新します。

サービスレベルマネージャ

サービスレベル管理

  • SLA評価期間終了時に実際の評価と合意済みのサービスレベルを比較するためにレポートを生成します。

  • SLA評価期間中に違反があった各SLTについて、違反の理由、サービスオーナーの対応、方策を見つけて再び違反がないようにします。

  • 顧客担当者へのSLA評価レポートには、事前に改善提案を追加します。

  • SLA評価期間終了後に、実際の評価と合意済みのサービスレベルとを、顧客担当者と確認します。

  • 顧客担当者から受け取ったリクエストは、サービス管理アプリケーションでサポートリクエストとして登録し、その後サービスデスクに渡します。

  • SLAレビュー会議の議事録をとり、配信します。

サービスオーナー

リクエスト管理

エスカレーション済みのインシデントが緊急の変更の実装継続サイトで影響を受けるサービスの復旧、または、リクエスト管理プロセスのインシデント解決の継続により、解決する必要があるかどうかを決定します。

スタッフ

変更管理

  • リスクとインパクトの作業依頼について、回答および該当情報とステータス変更による更新を行います。

  • 実装作業依頼を完成させ、該当情報とステータス変更による更新を行います。

リクエスト管理

  • サポートリクエストを解決します。

  • 関連する情報とステータスを変更して、サポートリクエストを更新します。

  • 変更管理プロセスを介して解決策を実装する必要がある場合、影響を受けるサービスのサービスオーナーにインシデントをエスカレートします。

問題管理

  • 問題の回避策を提案します。

  • 問題の特定のために根本原因をを確定します。

  • 問題に対する構造的な解決策を提案します。

  • 変更管理が不要な場合、問題に対する構造的な解決策を実装します。

  • 関連情報とステータス変更による問題の更新を行います。

ナレッジ管理

ナレッジ記事を提案します。

ナレッジマネージャ

ナレッジ管理

  • ナレッジ記事をレビューします。

  • 必要に応じてナレッジ記事を変更します。

  • ナレッジ記事を公開します。

  • 無効になっている記事の公開を終了します。
