手順2、CIの受領


変更管理またはリクエスト管理からのタスクがCMDBへ新しいCIを追加しない場合、構成管理者は直接手順4、CIの更新へ進みます。

一方、正式な購入申請が送信されなかった（たとえば社内で開発されたソフトウェアの場合などの）1つ以上のCIの登録を、タスクがリクエストする場合、または1つ以上の新しいCIが注文された場合、構成管理者はまず、納品後のCIを確認します。構成管理者は、正しいハードウェア、ソフトウェアおよびソフトウェアライセンス証明書が受け取られたか、不足しているアイテムはないか、CIが損傷していないかを確認するためにこれを行います。納品が一部のみだった場合は、構成管理者はタスクを更新することにより、サプライヤ、続いて変更コーディネータに遅延を通知します。

逆に、CIが良好な状態で受け取られた場合、構成管理者はサービス管理アプリケーションでサプライヤの連絡先の詳細が最新であるか確認します。CIのサプライヤがまだ登録されていない場合、または連絡先の詳細が最新でない場合、構成管理者はサプライヤ情報が手順3、サプライヤ情報の保守で登録または更新されることを確認します。

サプライヤ情報を更新または登録した後、CIが既に手順1、CIの申請で既に登録されている場合、構成管理者は新しいCIのステータスを更新し、不足している情報があれば追加します（例えばシリアルナンバーなど）。ただし、新しいCIについて正式の購入申請が送信されていない場合、構成管理者は、CMDBに新しいCIを登録します。構成管理者は、新しいCIの全属性、新しいCIと他のCIのリンク、サービスとユーザーのリンクが正しく登録されていることを確認します。

新しいCIがCMDBに適切に登録されていることを確認した後、構成管理者は、各CIが適用可能になった際にCIコードのラベルを貼付します。
