手順2、リリース定義


次のサービスのCABミーティングが予定されている場合、リリースコーディネータはサービスのCABスタッフメンバーに招待状を送信します。招待状の中で、リリースコーディネータは、次のCABミーティングでレビューする変更リクエストの送信期限日を指定します。リクエストの送信期限日を過ぎたら、リリースコーディネータは次のCABミーティングでレビューする変更リストのリストを最終決定し、CABスタッフメンバーがミーティング以前に内部的に話ができるよう、このリストを送信します。

最終決定された変更リクエストのリストは、ミーティング中にレビューされます。すべての変更リクエストのレビューが終わったら、各CABスタッフメンバーは、どの変更リクエストを却下して、どの変更リクエストを次のリリースにいれるかをリリースコーディネータに知らせます。変更リクエストは、CABスタッフメンバーの3分の2以上の同意があった場合に、却下されます。同様に、変更リクエストは、CABスタッフメンバーの3分の2以上の要求があった場合に、次にリリースされます。却下されず、次にリリースされない変更リクエストは、CABの次のミーティング中に再度レビューされます。CABミーティングの終了前に、CABスタッフメンバーは、次のミーティングの日付、時間、場所について合意します。

CABミーティング後リリースコーディネータは、変更リクエストを完了することで、却下された変更リクエストのリクエスト者が、通知されることを確認します。これを行ったことで、リリースコーディネータはCABミーティングのメモを発行し、新しいリリースを登録します。リリースコーディネータは、変更とリリースの調整をリンクし、このリリースに含める変更リクエストをこの変更にリンクします。

その後でリリースコーディネータは、リリースが実装されるサービスの変更コーディネータと、変更しなければならないサービスをサポートしている変更コーディネータとともに、これらの変更リクエストをレビューします。協力しながら、さまざまな変更リクエストの要件を分類し、高レベルな実装計画のドラフトを作成します。この計画では、各変更の所要時間と、変更間の依存関係を示します。
