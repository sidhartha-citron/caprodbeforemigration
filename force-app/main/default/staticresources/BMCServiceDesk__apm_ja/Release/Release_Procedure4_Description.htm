手順4、リリースの調整


リリースコーディネータは各変更コーディネータ用に、変更コーディネータが担当するリリースの要件を詳しく説明するドキュメントを準備します。リリースコーディネータは変更コーディネータに要件ドキュメントを送信して、要求された変更が登録され、リリースにリンクされたことを確認します。

変更コーディネータが、リリースの変更が完了したことを リリースコーディネータに通知した場合、リリースコーディネータは、リリースに短い更新情報を入力します。

変更の実装に失敗した場合、リリースコーディネータはリリースにリンクされた変更リクエストのコーディネータとともに、その（部分的な）失敗が、リリースの実装全体にどの程度の影響を与えるものかを判断します。1つ以上の変更を追加して、リリースの要件のすべてまたはほとんどを満たすことが可能である場合、または1つ以上の変更を追加して、リリースのロールバックの完了が確認できる場合、リリースコーディネータは適切な変更コーディネータに渡す前に、追加の変更の要件を文書にします。

変更コーディネータに要求されていた最後の変更の実装が完了したら、リリースコーディネータは、リクエスト者がリリースの実装によって要件が満たされたかどうかを通知されていることを確認します。リリースがCABによって定義されている場合、リリースコーディネータは、次にCABミーティングでレビューされる変更リクエストのプレースホルダーとして動作する、要件が満たされていない変更リクエストを移動します。リリースコーディネータは、CABスタッフメンバー（リリースがかれらによって定義されている場合）と、ビジネスケースの承認者（リリースのために予算を準備しなければならなかった場合）にも通知を行います。

リリースのリクエスト者と承認者が更新されるため、リリースコーディネータは実装後のミーティングの準備と開催を行います。ミーティング中、リリースコーディネータはリリースが実装されたサービスのサービスオーナー、および1つまたは複数のリリースの変更リクエストを調整した変更コーディネータと共に、リリースのライフサイクル全体を確認します。ミーティング後、リリースコーディネータは、ミーティング中に出席者が同意した改善案と決定事項を発行します。最後に、リリースコーディネータはリリースを終了します。
