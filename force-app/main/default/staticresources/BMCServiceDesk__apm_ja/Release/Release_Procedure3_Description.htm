手順3、業務上の変更理由


承認されたビジネスケースがリリースに必要な場合、リリースコーディネータは、リリースの変更リクエストがリンクされている変更の一部である、業務上の変更理由の処理順序をキャンセルします。一方で、サービスオーナーがすでに利用可能である予算を使って次のリリースが実装できない場合、リリースコーディネータはビジネスケースを作成し、リリースの理由を指定します。ビジネスケースを作成したら、リリースコーディネータは、次のリリースの実装に予算が利用可能であることを確認するために、ビジネスケースの承認を要求します。

ビジネスケースが却下された場合、リリースコーディネータはこのビジネスケースが承認されるように調整します。ビジネスケースの承認を得られない場合、リリースコーディネータはリリースに対するビジネスケースが却下されたことをこのリリースのリクエスト者に知らせます。リリースがCABよって定義されている場合、リリースコーディネータは、次にCABミーティングでレビューされる変更リクエストのプレースホルダーとして動作する、リリースにリンクされた変更リクエストを移動します。リリースを終了する前に、リリースコーディネータは、CABスタッフメンバーにも連絡を取り、リリースのビジネスケースが却下されたことと、次のCABミーティングで進めることもできることを伝えます。
