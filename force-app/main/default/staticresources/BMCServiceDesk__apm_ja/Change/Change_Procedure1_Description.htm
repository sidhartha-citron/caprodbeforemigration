手順1、変更の登録

変更コーディネータは、サービスの変更リクエストを受け、変更を調整します。変更リクエストは、次のソースが作成元となります:

  • リクエストの形式でのリクエスト管理

  • ステータスが「変更リクエスト済み」の問題の形式での問題管理

  • 要件の詳細を含むドキュメントの形式でのリリース管理。

変更リクエストを受け取ると、変更コーディネータは、登録済みの変更に追加できるかどうかを判断します。その場合、変更コーディネータは、変更リクエストを既存の変更へリンクします。

変更リクエストが、標準の変更の実装を要求するものである場合、変更コーディネータは、標準の変更に作成された変更テンプレートを用いて新規変更を登録し、それにリクエストをリンクします。変更コーディネータは、引き続きテンプレートからコピーされたワークフローに従い、変更を実装します。

変更リクエストが標準の変更のリクエストではない場合、変更コーディネータは、リクエストが社内標準またはポリシーと競合していないか検証します。競合する場合、変更リクエストは却下され、リクエスト者には関連する社内標準またはポリシーが通知されます。

競合がない場合、変更コーディネータは、変更の実装は変更管理によって調整する必要があるか判断します。この調整は、実装により次のような影響があるときのみ、必要です:

  • サービス時間内にサービスが利用できない、またはサービスが低下する

  • サービスの機能が変更になる

  • CMDBの更新が必要になる

変更の実装に変更管理が不要である場合は、変更コーディネータは変更リクエストを却下し、リクエスト者に変更管理が必要ないことを通知します。

変更管理がリクエストされた変更の実装に必要な場合、変更コーディネータは変更リクエストをリリース管理へ渡す必要があるか判断します。リリース管理は、変更リクエストが問題の防止または修正を要求するものでない場合は必要ありません。ただし次の条件によります:

  • 1名の変更コーディネータが変更の実装を行う

  • 実装によりサービスの機能が変更されない

  • 実装の完了まで追加資金が必要ない

リリース管理が必要でない場合、または変更リクエストの作成元がリリース管理である場合、変更コーディネータは新規変更を登録し、リクエストの要件をリンクします。その他すべての変更リクエストはリリース管理に割り当てられます。
