手順7:インシデントのエスカレーション処理


スタッフまたはサービスデスクアナリストがインシデントをエスカレーションした後、影響を受けるサービスのオーナーは、現状を把握しインシデントの解決方法を決定するため、そのインシデントを処理したスタッフと話し合います。

インシデントがメジャーインシデントの場合、サービスオーナーはメジャーインシデントチームを有効化します。メジャーインシデントチームリーダーは、解決方法を調整する責任を担います。

インシデントを解決するために、継続サイトによる影響を受けるサービスの復旧が最も有効で信頼性の高い方法である場合、サービスオーナーはインシデントを勤務中のマネージャにエスカレートして、サービスの復旧を開始します。

多くの場合は、継続サイトによるサービスの復旧ではインシデントは修正されません（たとえばバグが原因の場合）。サービスの現在のインフラで修正を実装した方が、有効で信頼性が高い可能性があります。そのような場合、サービスオーナーは、変更管理によってインシデントの解決方法を調整すべきか決定しなければなりません。インシデントの解決方法によって次のような状況が引き起こされる場合は、変更管理が必要になります。

  • サービス時間内にサービスが利用できない、またはサービスが低下する
  • 機能が変更になる
  • CMDBの更新が必要になる　

変更管理が不要な場合、サービスオーナーは最も適切なスタッフに、リクエスト管理プロセス内でこのインシデントの解決作業を続けさせます。

変更管理が必要な場合、サービスオーナーは変更実装の失敗原因となりうるリスクと、実装による顧客へのインパクトを把握するため、スタッフに問い合わせを行います。変更実装によるリスクとインパクトを、許容レベル内に抑える最善の方法を算出します。変更実装の方法が決まると、サービスオーナーはこの変更を緊急の変更として実装するようスタッフに依頼します。

影響を受けるサービスのオーナーが不在の場合、サービスオーナーの役割は勤務中のマネージャが担当します。
